# v1.1 PostgreSQL HA Mode — Technická analýza

## Popis

Přidání PostgreSQL jako alternativního databázového backendu vedle SQLite. Umožní horizontální škálování a vysokou dostupnost (HA) pro produkční nasazení.

### Proč

- SQLite je single-writer — při vysokém concurrency (50+ paralelních requestů) dochází k write lock contention
- SQLite nepodporuje replikaci — single point of failure
- Enterprise nasazení vyžadují HA databázi
- `initial-analyse.md:218,478-482` definuje PostgreSQL jako HA backend

### Out of Scope

- Migrace dat SQLite → PostgreSQL (manuální nástroj, budoucí)
- Connection pooling (PgBouncer) — dokumentace doporučení, ne implementace
- Read replicas — budoucí

## Návrh řešení

### Architektura

```
config.yaml
  database:
    backend: "postgres"    ←── nebo "sqlite"
    postgres:
      dsn: "postgres://..."
    sqlite:
      path: "/var/lib/..."
         │
         ▼
config.InitDB(cfg.Database)
         │
    ┌────┴────┐
    │ sqlite  │ postgres │
    │ Open()  │ Open()   │
    │ PRAGMAs │ (none)   │
    │ migrate │ migrate  │
    └─────────┘──────────┘
         │
         ▼
    *sqlx.DB (same interface everywhere)
```

**Klíčová rozhodnutí:**

1. **`sqlx.DB` interface zůstává** — veškerý kód používá `*sqlx.DB`, žádné změny v adaptérech, API, scheduleru.
2. **Dual-dialect migrace** — každá migrace má SQLite a PostgreSQL verzi. Migration runner detekuje backend a spustí správnou variantu.
3. **`?` placeholder → thin DB wrapper s automatickým Rebind** — Všechen existující kód používá `db.Exec(query, args...)` přímo. `sqlx` NE-rebinduje automaticky — je nutné volat `db.Rebind(query)` před každým voláním. Pouhý helper na `Exec` nestačí (nepokrývá `Get`, `Select`, `Query`, `QueryRow`). **Řešení:** Vytvořit thin wrapper struct `type GateDB struct { *sqlx.DB }` s přetíženými metodami `Exec`, `Get`, `Select`, `Query`, `QueryRow` které interně volají `Rebind()`. Všechen kód pak používá `*GateDB` místo `*sqlx.DB`. Toto je **velký refaktor** — desítky call-sites v adaptérech, API, policy, scheduler.
4. **`INSERT OR REPLACE` → `ON CONFLICT DO UPDATE`** — refaktor. **Pozor na sémantický rozdíl:** `INSERT OR REPLACE` v SQLite DELETUJE a RE-INSERTUJE řádek (resetuje ROWID, triggeruje DELETE triggers). `ON CONFLICT DO UPDATE` updatuje in-place. Pro `artifacts` a `artifact_status` tabulky (s FK referencemi) je sémanticky bezpečnější. **Kandidát na ADR:** Toto a `SERIAL PRIMARY KEY` jsou vendor extensions, nikoli ANSI SQL — to je odchylka od CLAUDE.md pravidla 'no vendor extensions', ale pragmaticky nutná.
5. **PRAGMAs podmíněné** — pouze pro SQLite backend.

### Změny v `db.go`

```go
// internal/config/db.go

func InitDB(cfg DatabaseConfig) (*sqlx.DB, error) {
    switch cfg.Backend {
    case "postgres":
        return initPostgres(cfg.Postgres)
    default:
        return initSQLite(cfg.SQLite)
    }
}

func initSQLite(cfg SQLiteConfig) (*sqlx.DB, error) {
    // existující logika — Open, PRAGMAs, migrate
}

func initPostgres(cfg PostgresConfig) (*sqlx.DB, error) {
    db, err := sqlx.Open("postgres", cfg.DSN)
    if err != nil {
        return nil, fmt.Errorf("config: connecting to postgres: %w", err)
    }
    db.SetMaxOpenConns(cfg.MaxOpenConns)    // default 25
    db.SetMaxIdleConns(cfg.MaxIdleConns)    // default 5
    db.SetConnMaxLifetime(cfg.ConnMaxLifetime) // default 5m
    if err := db.Ping(); err != nil {
        return nil, fmt.Errorf("config: pinging postgres: %w", err)
    }
    if err := runMigrations(db, "postgres"); err != nil {
        return nil, err
    }
    return db, nil
}
```

### Konfigurace

```go
type DatabaseConfig struct {
    Backend  string         `mapstructure:"backend"`   // "sqlite" | "postgres"
    SQLite   SQLiteConfig   `mapstructure:"sqlite"`
    Postgres PostgresConfig `mapstructure:"postgres"`
}

type PostgresConfig struct {
    DSN             string `mapstructure:"dsn"`               // connection string
    MaxOpenConns    int    `mapstructure:"max_open_conns"`    // default 25
    MaxIdleConns    int    `mapstructure:"max_idle_conns"`    // default 5
    ConnMaxLifetime string `mapstructure:"conn_max_lifetime"` // default "5m"
}
```

YAML:
```yaml
database:
  backend: "postgres"
  postgres:
    # DSN NIKDY neuvádějte s credentials v config souboru!
    # Použijte env var: SGW_DATABASE_POSTGRES_DSN
    dsn: ""  # povinné, typicky z env var
    require_tls: true  # odmítne sslmode=disable/prefer (default true)
    max_open_conns: 25
    max_idle_conns: 5
    conn_max_lifetime: "5m"
```

**DSN bezpečnost:**
- Config NESMÍ obsahovat default DSN s credentials (prevence commitnutí do VCS)
- Startup WARNING pokud DSN obsahuje heslo a bylo načteno z config souboru (ne z env var)
- DSN se NIKDY neloguje (zerolog scrubbing) — invariant #3
- `require_tls: true` (default) odmítne `sslmode=disable` a `sslmode=prefer` při startu

**Bezpečnost:** DSN obsahuje credentials. Podporovat env var override přes viper: `SGW_DATABASE_POSTGRES_DSN`.

### Dual-dialect migrace

```
internal/config/migrations/
├── sqlite/
│   ├── 001_init.sql
│   ├── 002_policy_overrides.sql
│   ├── 003_docker_registry.sql
│   ├── 004_docker_tags.sql
│   └── ... (budoucí)
└── postgres/
    ├── 001_init.sql
    ├── 002_policy_overrides.sql
    ├── 003_docker_registry.sql
    ├── 004_docker_tags.sql
    └── ... (budoucí)
```

Rozdíly:
- `INTEGER PRIMARY KEY AUTOINCREMENT` → `SERIAL PRIMARY KEY`
- `datetime('now', '+24 hours')` → `NOW() + INTERVAL '24 hours'`
- `PRAGMA` příkazy → odstraněny
- `DATETIME` typ → PostgreSQL nemá `DATETIME`, použít `TIMESTAMP` (nebo `TIMESTAMPTZ`)
- `schema_migrations` tabulka samotná musí být dual-dialect kompatibilní

**Embed.FS restrukturace:** Stávající `//go:embed migrations/*.sql` v `db.go:13` musí být změněno na:
```go
//go:embed migrations/sqlite/*.sql
var sqliteMigrations embed.FS

//go:embed migrations/postgres/*.sql
var postgresMigrations embed.FS
```
Migration runner vybírá správný embed.FS podle backendu. Migrace MUSÍ zůstat embedded (compiletime), nikdy čtené z disku za runtime (bezpečnost).

### SQL refaktoring — `INSERT OR REPLACE`

`internal/adapter/base.go:152,170` — `INSERT OR REPLACE INTO artifacts` / `artifact_status`:

Refaktor na:
```sql
INSERT INTO artifacts (...) VALUES (...)
ON CONFLICT (id) DO UPDATE SET
    upstream_url = EXCLUDED.upstream_url,
    sha256 = EXCLUDED.sha256,
    ...
```

Funguje na SQLite 3.24+ (2018) i PostgreSQL 9.5+ (2016). Zpětně kompatibilní.

### Placeholder rebinding — GateDB wrapper

```go
// internal/config/db.go

// GateDB wraps sqlx.DB and auto-rebinds all queries.
// All code MUST use *GateDB instead of *sqlx.DB.
type GateDB struct {
    *sqlx.DB
}

func (db *GateDB) Exec(query string, args ...any) (sql.Result, error) {
    return db.DB.Exec(db.Rebind(query), args...)
}

func (db *GateDB) Get(dest any, query string, args ...any) error {
    return db.DB.Get(dest, db.Rebind(query), args...)
}

func (db *GateDB) Select(dest any, query string, args ...any) error {
    return db.DB.Select(dest, db.Rebind(query), args...)
}

// + Query, QueryRow, QueryRowContext, ExecContext, etc.
```

**Blast radius:** Všechny soubory používající `*sqlx.DB` musí být aktualizovány na `*GateDB`:

## Akceptační kritéria

1. `database.backend: "postgres"` → aplikace se připojí k PostgreSQL a migruje schema.
2. `database.backend: "sqlite"` → chování identické jako dnes.
3. Všechny existující testy projdou s oběma backendy.
4. Žádný `INSERT OR REPLACE` v codebase — nahrazeno `ON CONFLICT DO UPDATE`.
5. PostgreSQL migrace vytvoří identické schéma jako SQLite (modulo syntax).
6. Connection pooling konfigurovatelný (max_open_conns, max_idle_conns, conn_max_lifetime).
7. DSN může být poskytnut přes environment variable; config file hodnota je přepsatelná.
8. Integration test suite běží proti oběma backendům v CI s identickými výsledky.
9. `sslmode=disable` v DSN je odmítnut pokud `require_tls: true` (default).
10. PostgreSQL používá `READ COMMITTED` isolation level (dokumentovat rozdíl oproti SQLite SERIALIZABLE).

## Rizika a mitigace

| Riziko | Dopad | Pravděpodobnost | Mitigace |
|--------|-------|-----------------|----------|
| Skrytý SQLite-only SQL | Vysoký — runtime error na PostgreSQL | Střední | Integration test suite s PostgreSQL |
| `?` vs `$1` placeholder | Vysoký — broken queries | Střední | sqlx.Rebind(), CI test s oběma backendy |
| Migration divergence | Střední — schema drift | Nízká | CI check: obě migrace musí vytvořit kompatibilní schéma |
| DSN s credentials v config | Střední — security | Nízká | Env var override SGW_DATABASE_POSTGRES_DSN |
| GateDB refaktor — velký blast radius | Střední — mnoho souborů | Jistý | Mechanická změna; compiler odhalí chybějící call-sites |
| `DATETIME` vs `TIMESTAMP` v migracích | Střední — runtime error PG | Střední | Dual-dialect migrace pokrývá |
| Transaction isolation level rozdíl | Nízký — subtilní behavioral rozdíly | Nízká | Dokumentovat; PostgreSQL `READ COMMITTED` je standard |
| DSN s credentials commitnutý do VCS | Kritický | Střední | Config NESMÍ mít default DSN; startup warning |

## Poznámky

- **Vendor extensions (ADR kandidát):** `SERIAL PRIMARY KEY` (PG), `AUTOINCREMENT` (SQLite), `ON CONFLICT DO UPDATE` (obojí) jsou vendor extensions. CLAUDE.md říká 'no vendor extensions', ale tyto jsou pragmaticky nutné pro dual-backend support. Dokumentovat jako ADR.
- **CGO závislost:** SQLite vyžaduje CGO (`mattn/go-sqlite3`). PostgreSQL (`lib/pq`) je pure Go. Pro PostgreSQL-only nasazení by šlo odstranit CGO dependency — ale to je budoucí optimalizace.
- **Zpětná kompatibilita:** 100% — default backend zůstává `sqlite`.
- **Dokumentační deliverables:** `docs/index.md` update, `docs/configuration.md` (PostgreSQL setup guide), ADR pro vendor extensions, ADR pro GateDB wrapper pattern.
