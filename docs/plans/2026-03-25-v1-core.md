# Shieldoo Gate v1.0 Core — Technická analýza

## Popis

Implementace kompletního jádra Shieldoo Gate — open-source supply chain security proxy, který transparentně proxuje požadavky na balíčky, skenuje artefakty před doručením a blokuje škodlivý obsah v reálném čase.

### Proč

- **Supply chain útoky jsou reálná hrozba** — incident LiteLLM/Trivy (březen 2026) ukázal, že kompromitované balíčky mohou exfiltrovat SSH klíče a cloud credentials z milionů systémů
- **Neexistuje open-source řešení** — Harbor pokrývá pouze Docker, komerční řešení (Sonatype, JFrog) jsou drahá a uzavřená
- **Zero client-side configuration** — vývojáři jen přesměrují package manager na proxy, žádné změny v workflow
- **v1.0 je MVP** — minimální funkční sada pro reálné nasazení, hardening a enterprise features přijdou v dalších verzích

### Business hodnota

- Ochrana CI/CD pipeline a vývojářských stanic před kompromitovanými balíčky
- Jednotný nástroj pro PyPI, npm, Docker a NuGet
- Self-hostable, Apache 2.0 licence

### Scope boundaries

**V rozsahu v1.0:**
- PyPI, npm, Docker, NuGet protokolové adaptery
- GuardDog + Trivy + OSV externí skenery + built-in heuristiky
- Local filesystem cache s TTL a LRU eviction
- SQLite databáze
- Rule-based policy engine (allow/block/quarantine/warn)
- Community threat feed (read-only, HTTP polling)
- React admin dashboard (read-only, bez autentizace)
- Docker Compose deployment
- Manual rescan přes REST API

**Odloženo na v1.1** (viz [roadmap ve specifikaci](../initial-analyse.md#15-roadmap)):
- Rescan scheduler (`internal/scheduler/`) — automatické periodické přeskenování cachovaných artefaktů
- Alerts (Slack, webhook, email) — `internal/alert/`
- S3 / Azure Blob / GCS cache backends
- Maven, RubyGems, Go modules adaptery
- OIDC admin autentizace
- PostgreSQL HA mode
- Helm chart
- Dynamic sandbox execution (gVisor)

**Důsledky odloženého rescan scheduleru:** Pole `rescan_due_at` v `artifact_status` tabulce je v v1.0 vždy prázdné — připraveno pro v1.1. Nově objevené hrozby v threat feedu se projeví pouze při novém stažení artefaktu (cache miss nebo manuální rescan přes API). Admin může manuálně triggerovat rescan přes `POST /api/v1/artifacts/{id}/rescan`.

**Důsledky chybějící autentizace:** Admin API (port 8080) je v v1.0 **neautentizované**. Nasazení musí zajistit, že port 8080 je přístupný pouze z důvěryhodné sítě (firewall, reverse proxy s auth). Toto je dokumentováno jako riziko.

## Návrh řešení

### Architektura

```
┌─────────────────────────────────────────────────────────────────┐
│                        Shieldoo Gate                            │
│                                                                 │
│  cmd/shieldoo-gate/main.go                                      │
│    ├── config.Load()                                            │
│    ├── db.Init() (SQLite)                                       │
│    ├── cache.NewLocalStore()                                    │
│    ├── scanner.NewEngine(guarddog, trivy, osv, builtins)        │
│    ├── policy.NewEngine(rules)                                  │
│    ├── threatfeed.NewClient() (internal/threatfeed/)             │
│    ├── adapter.NewPyPI/NPM/Docker/NuGet(scanner, cache, policy) │
│    ├── api.NewServer(handlers)                                  │
│    └── chi.Router → mount all endpoints                         │
│                                                                 │
│  Request flow:                                                  │
│  Client → Adapter → Cache check                                 │
│    [hit]  → artifact_status check → QUARANTINED? → 403          │
│                                   → TTL expired? → treat as miss│
│                                   → CLEAN → Serve from cache    │
│    [miss] → Upstream download → Scan Engine (parallel)          │
│            → Policy evaluation → Cache write (if clean)         │
│            → Serve / Block (403) / Quarantine                   │
└─────────────────────────────────────────────────────────────────┘

Sidecar:
  scanner-bridge (Python/gRPC) ← GuardDog scanner
```

Klíčové architektonické principy:

1. **Každý adapter je samostatný HTTP handler** — implementuje `adapter.Adapter` interface, mountuje se na dedikovaný port
2. **Scanner Engine orchestruje paralelní skenování** — spouští všechny relevantní skenery pro daný ekosystém současně
3. **Policy Engine je čistý evaluátor** — přijme agregovaný výsledek skenů, vrátí akci (allow/block/quarantine/warn)
4. **Cache Store je transparentní** — adaptery nepracují přímo se souborovým systémem, ale přes `CacheStore` interface
5. **Fail-open design** — selhání skeneru nikdy neeskaluje na MALICIOUS, loguje se chyba a vrací CLEAN

### Databázové změny

Kompletní DDL pro SQLite (z technické specifikace):

| Tabulka | Sloupce | Popis |
|---------|---------|-------|
| `artifacts` | id (PK), ecosystem, name, version, upstream_url, sha256, size_bytes, cached_at, last_accessed_at, storage_path | Metadata cachovaných artefaktů |
| `scan_results` | id (PK, autoincrement), artifact_id (FK), scanned_at, scanner_name, scanner_version, verdict, confidence, findings_json, duration_ms | Historie skenů — jeden řádek per sken per skener |
| `artifact_status` | artifact_id (PK, FK), status, quarantine_reason, quarantined_at, released_at, rescan_due_at, last_scan_id (FK) | Denormalizovaný aktuální stav pro rychlý lookup. `PENDING_SCAN` status a `rescan_due_at` pole jsou v v1.0 nevyužité — připravené pro rescan scheduler v v1.1. |
| `audit_log` | id (PK, autoincrement), ts, event_type, artifact_id, client_ip, user_agent, reason, metadata_json | Append-only audit log — NIKDY UPDATE/DELETE |
| `threat_feed` | sha256 (PK), ecosystem, package_name, version, reported_at, source_url, iocs_json | Lokální kopie community threat feedu |

```sql
-- Indexy
CREATE INDEX idx_artifacts_ecosystem_name ON artifacts(ecosystem, name);
CREATE INDEX idx_scan_results_artifact ON scan_results(artifact_id);
CREATE INDEX idx_audit_log_ts ON audit_log(ts);
CREATE INDEX idx_threat_feed_ecosystem ON threat_feed(ecosystem, package_name);
```

Migrace: jednoduchý SQL soubor spouštěný při startu (`internal/config/migrations/001_init.sql`). Pro v1.0 není potřeba migrační framework — SQLite databáze se vytváří při prvním spuštění.

**SQLite PRAGMAs** — nastavit při každém otevření spojení (ne v migraci):
```sql
PRAGMA journal_mode=WAL;      -- write-ahead logging pro concurrent reads
PRAGMA foreign_keys=ON;        -- SQLite má FK defaultně vypnuté
PRAGMA busy_timeout=5000;      -- 5s retry pro SQLITE_BUSY
```

#### internal/adapter/

```go
// interface.go — již definováno v technické specifikaci
type Adapter interface {
    Ecosystem() Ecosystem
    ServeHTTP(w http.ResponseWriter, r *http.Request)
    HealthCheck(ctx context.Context) error
}
```

Každý adapter:
1. Parsuje požadavek dle nativního protokolu, **validuje package name/version** (whitelist povolených znaků, ochrana proti path traversal v cache paths)
2. Kontroluje cache — při hitu ověří `artifact_status.status`:
   - `QUARANTINED` → HTTP 403 s JSON error body (viz [sekce 8 specifikace](../initial-analyse.md#8-rescan-scheduler))
   - TTL expired → treat jako cache miss (re-download + re-scan)
   - `CLEAN` a v TTL → serve přímo z cache
3. Při cache miss stahuje z upstreamu do temp souboru
4. Volá `scanner.Engine.ScanAll()`
5. Volá `policy.Engine.Evaluate()`
6. Podle výsledku:
   - `allow` → DB write (artifacts + artifact_status) → cache write → serve, audit log `SERVED`
   - `block` → HTTP 403, audit log `BLOCKED`
   - `quarantine` → DB write (status=QUARANTINED) → cache write (soubor uložen ale neservírován), audit log `QUARANTINED`
   - `warn` → serve artefakt + HTTP header `X-Shieldoo-Warning: <reason>`, audit log `WARNED`
7. Zapisuje do audit logu (async přes buffered channel)

**Transakční hranice:** DB write (artifacts + artifact_status) musí proběhnout před cache write na filesystem. Při selhání DB writeu se artefakt neukládá do cache. Při selhání cache writeu se DB záznam označí jako chybový. Cache write používá atomic rename (`os.Rename` z temp souboru) pro konzistenci.

## Rizika a mitigace

| Riziko | Dopad | Pravděpodobnost | Mitigace |
|--------|-------|-----------------|----------|
| GuardDog Python bridge přidává latenci a komplexitu | Střední — pomalejší skeny, extra proces ke správě | Vysoká | gRPC over Unix socket minimalizuje overhead; health check + automatic restart; timeout na scan operace |
| Trivy binary nemusí být dostupný na všech platformách | Střední — Docker adapter bez CVE skenování | Nízká | Trivy je volitelný skener; pokud chybí, loguje warning a pokračuje bez něj |
| OCI Distribution Spec je komplexní (manifest lists, multi-arch) | Vysoký — Docker adapter nemusí fungovat se všemi registry | Střední | Využití `go-containerregistry` knihovny, která většinu komplexity abstrahuje; focus na Docker Hub + GHCR v v1.0 |
| SQLite performance pod zátěží (write contention) | Střední — pomalé logování při vysokém trafficu | Nízká pro v1.0 | WAL mode; batch writes pro audit log; PostgreSQL je v roadmapě pro v1.1 |
| Scanner false positives blokují legitimní balíčky | Vysoký — vývojáři nemohou pracovat | Střední | Allowlist v policy engine; `warn` akce místo `block` pro nízkou confidence; minimum_confidence threshold |
| First-request latence (download + scan) | Střední — pomalý UX při cache miss | Vysoká | Očekávané chování, dokumentované; warm cache je rychlý; background pre-fetch v budoucnu |
| Threat feed server nedostupný | Nízký — lokální kopie feedu stále funguje | Nízká | Graceful degradation; lokální DB kopie; logování warning |
| CGO dependency (go-sqlite3) komplikuje cross-compilation | Nízký — build jen pro Linux target | Nízká | Docker build řeší; pro lokální dev stačí matching OS |
| Admin API bez autentizace v v1.0 | Vysoký — kdokoliv může release quarantined artefakty | Střední | Port 8080 musí být za firewallem / reverse proxy s auth; OIDC auth v roadmapě pro v1.1 |
| Fail-open při výpadku všech skenerů | Vysoký — artefakty procházejí bez skenování | Nízká | Scanner health status viditelný v Dashboard UI (prominentní warning); health endpoint pro monitoring; `scanner_errors_total` Prometheus metrika |
| Path traversal přes malicious package names | Vysoký — přístup k souborům mimo cache | Nízká | Input validace whitelist znaků v package name/version ve všech adapterech |

## Poznámky

### Idempotence
- Opakovaný download stejného artefaktu vrací cache hit
- Opakovaný scan stejného artefaktu aktualizuje `scan_results`, ale nemění `artifact_status` pokud se verdict nezměnil
- DB migrace je idempotentní (`CREATE TABLE IF NOT EXISTS`)

### Edge cases
- **Artifact stahování selhalo** — vrátit 502 Bad Gateway, neukládat do cache, logovat error
- **Upstream nedostupný, artefakt v cache** — pokud je cached copy CLEAN (i když TTL expired), serve stale s `X-Shieldoo-Warning: stale-cache` headerem. Pokud není v cache, vrátit 502.
- **Všechny skenery selhaly** — fail open (CLEAN), logovat všechny chyby, pokračovat s doručením. Dashboard UI zobrazí prominentní warning.
- **Upstream vrací redirect** — adapter musí followovat redirecty (standardní chování HTTP klienta)
- **Velký Docker image (multi-GB)** — streaming download/upload, nekopírovat celý image do paměti. Trivy skenuje z temp souboru na disku, ne z paměti.
- **Concurrent requests na stejný uncached artefakt** — v v1.0 se každý request zpracuje nezávisle (duplicitní download + scan). Známá optimalizace pro budoucnost: `singleflight` pattern pro deduplikaci.
- **Concurrent cache write** — atomic rename (`os.Rename` z temp souboru) pro konzistenci
- **Cache plný** — LRU eviction dle `last_accessed_at`. Quarantined artefakty se neevictují automaticky.
- **Cache TTL expired** — treat jako cache miss, re-download z upstreamu + re-scan
- **SQLite write contention** — WAL mode, `PRAGMA busy_timeout=5000`, retry s backoff pro SQLITE_BUSY
- **Path traversal v package name** — whitelist validace znaků (`[a-zA-Z0-9._-]`) před konstrukcí cache paths
- **Quarantine release** — admin release nastaví status=CLEAN + released_at timestamp + audit log RELEASED. Artefakt může být znovu quarantined při manuálním rescanu.

### Výkonnostní úvahy
- Cache hit: <5ms response time (čtení z disku + DB lookup artifact_status)
- Cache miss: upstream download time + scan time (řádově sekundy až minuty pro velké Docker images)
- Scanner paralelizace zkracuje celkový scan time na maximum nejpomalejšího skeneru
- Audit log zápisy by neměly blokovat response — async write do buffered channel. Channel musí být drained při graceful shutdown (viz Fáze 8).
- OSV API má rate limity — respektovat `Retry-After` headers, cachovat OSV responses
