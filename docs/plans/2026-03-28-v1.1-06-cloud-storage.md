# v1.1 Cloud Storage Backends (S3 / Azure Blob / GCS) — Technická analýza

## Popis

Implementace S3, Azure Blob Storage a GCS backends pro `CacheStore` interface. Umožní ukládání artefaktů v cloudovém object storage místo lokálního filesystému.

### Proč

- Lokální filesystem nelze sdílet mezi více instancemi (HA deployment)
- Object storage nabízí prakticky neomezenou kapacitu, durabilitu, replikaci
- Kubernetes nasazení preferuje stateless pody s externím storage
- `initial-analyse.md:460-470` definuje S3/Azure Blob config

### Out of Scope

- Encryption at rest enforcement — delegováno na storage backend, ale dokumentovat jako deployment requirement (SSE-S3/SSE-KMS pro S3, default encryption pro Azure/GCS).
- Cross-region replication
- Cache eviction policy (budoucí)
- **Poznámka:** GCS přidáno nad rámec původního spec (`initial-analyse.md:460` definuje pouze `local | s3 | azureblob`). GCS doplněno pro pokrytí všech 3 major cloud providers.

## Aktuální stav

### CacheStore interface

`internal/cache/interface.go:23-29`:
```go
type CacheStore interface {
    Get(ctx context.Context, artifactID string) (localPath string, err error)
    Put(ctx context.Context, artifact scanner.Artifact, localPath string) error
    Delete(ctx context.Context, artifactID string) error
    List(ctx context.Context, filter CacheFilter) ([]string, error)
    Stats(ctx context.Context) (CacheStats, error)
}
```

**Kritický design problém:** `Get()` vrací `localPath string` — předpokládá lokální filesystem. Pro cloud storage je nutné stáhnout blob do temp souboru a vrátit jeho cestu. Toto je funkční ale neefektivní pro velké artefakty.

### Local implementace

`internal/cache/local/local.go:24-237`:
- Layout: `{basePath}/{ecosystem}/{name}/{version}/{filename}`
- `Get()` (řádky 99-120): hledá soubor na filesystem, vrací cestu
- `Put()` (řádky 124-143): kopíruje soubor do cache adresáře
- `Stats()` (řádky 206-237): walkuje filesystem, počítá velikosti

### Použití v adaptérech

Všechny adaptéry používají `cache.CacheStore` identicky:
- `cache.Get(ctx, artifactID)` → `localPath` → `http.ServeFile(w, r, localPath)`
- `cache.Put(ctx, scanArtifact, tmpPath)`
- `cache.Delete(ctx, artifactID)`

| Aspekt | Současný stav | Navrhovaný stav |
|--------|--------------|-----------------|
| Backend | Pouze local filesystem | Local + S3 + Azure Blob + GCS |
| Get() return | localPath (filesystem) | localPath (temp file pro cloud) |
| Interface | Beze změny | Beze změny (kompatibilní) |
| Config | `cache.backend: "local"` | + `"s3"`, `"azureblob"`, `"gcs"` |

## Návrh řešení

### Architektura

```
CacheStore interface (beze změny)
    │
    ├── local.LocalCacheStore     (existující)
    ├── s3.S3CacheStore           (nový)
    ├── azureblob.AzureBlobStore  (nový)
    └── gcs.GCSCacheStore         (nový)
```

Všechny cloud backendy sdílejí pattern:
- `Put()`: upload soubor do bucketu s klíčem `{ecosystem}/{name}/{version}/{sha256}`
- `Get()`: download blob do temp souboru, vrátit temp path
- `Delete()`: smazat blob
- `List()`: list keys s prefixem
- `Stats()`: agregace velikostí (list + metadata)

### Integrity verification po cloud download

**POVINNÉ:** Po stažení blobu z cloud storage, ověřit `SHA256(downloaded_file) == artifacts.sha256` z databáze. Pokud se neshoduje → odmítnout, logovat CRITICAL error, alert. Toto chrání proti:
- Kompromitovanému cloud bucketu
- Storage corruption
- Man-in-the-middle na cloud API

Bez tohoto ověření je porušen bezpečnostní invariant #2 ("Never trust artifact content before scan completes") — artefakt byl skenován před uploadem do cloudu, ale stažený obsah není verifikován.

### Stats() optimalizace pro cloud

Cloud `Stats()` vyžaduje list všech objektů v bucketu (S3: 1000 keys per page). Pro 100k+ artefaktů je to drahé (latence + API cost). **Řešení:**
- Cache stats v paměti, refresh každých 5 minut v background goroutine
- Nebo použít bucket-level metriky z cloud providera (S3 CloudWatch, GCS metrics)

### Konfigurace

```go
type CacheConfig struct {
    Backend   string           `mapstructure:"backend"` // "local"|"s3"|"azureblob"|"gcs"
    Local     LocalCacheConfig `mapstructure:"local"`
    S3        S3CacheConfig    `mapstructure:"s3"`
    AzureBlob AzureBlobConfig  `mapstructure:"azure_blob"`
    GCS       GCSCacheConfig   `mapstructure:"gcs"`
    TTL       TTLConfig        `mapstructure:"ttl"`
}

type S3CacheConfig struct {
    Bucket   string `mapstructure:"bucket"`
    Region   string `mapstructure:"region"`
    Endpoint string `mapstructure:"endpoint"` // pro MinIO
    Prefix   string `mapstructure:"prefix"`   // optional key prefix
}

type AzureBlobConfig struct {
    AccountName   string `mapstructure:"account_name"`
    ContainerName string `mapstructure:"container_name"`
    // Auth via AZURE_STORAGE_KEY env var nebo managed identity
}

type GCSCacheConfig struct {
    Bucket string `mapstructure:"bucket"`
    // Auth via GOOGLE_APPLICATION_CREDENTIALS nebo workload identity
}
```

**S3 endpoint SSRF ochrana:** `Endpoint` field umožňuje uživateli nasměrovat na libovolnou URL. Validovat při startu: logovat warning pro non-standard endpoint. Zvážit HTTPS-only enforcement (kromě localhost pro MinIO dev).

### Factory v main.go

```go
var cacheStore cache.CacheStore
switch cfg.Cache.Backend {
case "s3":
    cacheStore, err = s3cache.NewS3CacheStore(cfg.Cache.S3)
```

**Důvod:** Lokální cache `Get()` vrací trvalou cestu (soubor JE cache). Cloud `Get()` vrací temp soubor, který MUSÍ být odstraněn po servírování. Adaptéry dosud nevolají `os.Remove()` po `cache.Get()` — to je pro local backend korektní, ale pro cloud backend způsobí temp file leak.

**Řešení:** Cloud `Get()` vrátí wrapper s cleanup:
```go
type CacheResult struct {
    Path      string
    NeedClean bool  // true pro cloud backends
}
```
Nebo jednodušeji: cloud Get() registruje background goroutine pro cleanup po 5 minutách. Adaptéry se nemění, ale cloud backend interně sleduje vrácené temp files a periodicky je čistí.

## Akceptační kritéria

1. `cache.backend: "s3"` → artefakty se ukládají/čtou z S3 bucketu.
2. `cache.backend: "local"` → chování identické jako dnes.
3. `Get()` pro cloud backendy vrací cestu k temp souboru. Temp files jsou automaticky odstraněny do 5 minut (background cleanup).
4. MinIO kompatibilita (S3 s custom `endpoint`).
5. Interface compile-time checks: `var _ cache.CacheStore = (*s3.S3CacheStore)(nil)`
6. Po cloud download: SHA256 integrity check proti DB. Mismatch = reject + CRITICAL alert.
7. Dva Shieldoo Gate instance sdílející stejný S3 bucket mohou concurrent Put/Get bez data corruption.
8. `Stats()` neblokuje — vrací cached výsledek (max 5 minut starý pro cloud backendy).
9. TTL eviction funguje pro cloud backendy stejně jako pro local (Delete() smaže blob).

## Rizika a mitigace

| Riziko | Dopad | Pravděpodobnost | Mitigace |
|--------|-------|-----------------|----------|
| Latence cloud Get() (download to temp) | Střední — pomalejší cache hits | Střední | Local temp cache; streaming response (budoucí) |
| Temp file cleanup leak | Střední — disk plný | Nízká | defer os.Remove(); periodic cleanup goroutine |
| SDK version sprawl (3 cloud SDKs) | Nízký — dependency bloat | N/A | Každý backend je optional; build tags (budoucí) |
| Cloud auth complexity | Střední — config chyby | Střední | Defaultovat na standard env vars (AWS_*, AZURE_*, GOOGLE_*) |
| Kompromitovaný cloud bucket | Kritický — servírování tampered artefaktů | Nízká | SHA256 integrity check po download (invariant #2) |
| Stats() list na 100k+ objektů | Střední — API cost + latence | Střední | In-memory stats cache s 5min refresh |
| S3 endpoint SSRF | Střední — internal network | Nízká | Warning log pro non-standard endpoint; HTTPS enforcement |

## Poznámky

- **Get() temp file pattern:** Cloud Get() stáhne blob do `os.CreateTemp()` a vrátí cestu. Caller (adaptér) musí defer remove. Toto je kompatibilní s existujícím interface ale neefektivní. Streaming Get() (vrátit `io.ReadCloser`) by byl optimálnější — ale vyžaduje interface change, kandidát na v1.2.
- **Auth:** Všechny SDK defaultují na standard env vars / IAM roles. Žádné custom auth v konfiguraci.
- **Cloud client lifecycle:** SDK klienty (S3, Azure, GCS) se vytvářejí jednou při startu a sdílejí se (connection pooling interně v SDK). Nikdy se nevytvářejí per-request.
- **Dokumentační deliverables:** `docs/index.md`, `docs/configuration.md`, `docs/deployment.md` (cloud storage setup per provider).
