# v1.1 Alerting System (Webhook / Slack / Email) — Technická analýza

## Popis

Implementace alertovacího systému pro Shieldoo Gate, který odesílá notifikace při bezpečnostních událostech (blokace, karanténa, změna digestu tagu, release z karantény). Systém podporuje tři kanály: generický webhook, Slack a email (SMTP).

### Proč

- Alerting je **prerekvizita** pro rescan scheduler (v1.1) a plnohodnotnou tag mutability detection — bez notifikací administrátor neví, že se něco stalo
- Současný stav: bezpečnostní události jsou pouze logovány (zerolog) a zapisovány do audit_log tabulky — žádný proaktivní push k operátorovi
- Docker sync service (`sync.go:242-251`) karanténuje artefakty bez jakékoli notifikace
- Analýza `initial-analyse.md:515-528` definuje alerting config, ale implementace chybí

## Aktuální stav

### Jak jsou události dnes zpracovány

Všechny bezpečnostní události procházejí dvěma cestami:
1. **Strukturované logy** (zerolog) — `log.Info()` / `log.Warn()` / `log.Error()`
2. **Audit log tabulka** — `adapter.WriteAuditLog()` v `internal/adapter/base.go:76-97`

Žádná třetí cesta (push notifikace) neexistuje.

### Architektonické poznatky

1. **WriteAuditLog je single choke-point** — všechny audit záznamy prochází `adapter.WriteAuditLog()` (`base.go:76-97`). To je ideální místo pro hook alertu.
2. **Docker sync nemá audit log** — sync.go volá `persistArtifact()` ale nikdy `WriteAuditLog()`. Toto je bug k opravě.
3. **Config nemá alerts sekci** — `internal/config/config.go` obsahuje pouze Server, Ports, Upstreams, Cache, Database, Scanners, Policy, ThreatFeed, Log.
4. **Adaptéry nemají přístup k alerteru** — konstruktory přijímají `(db, cache, scanEngine, policyEngine, upstream)`.

| Aspekt | Současný stav | Navrhovaný stav |
|--------|--------------|-----------------|
| Notifikace | Žádné | Webhook + Slack + Email |
| Audit log | Zápis do DB bez side-effectů | Zápis do DB + dispatch alertu |
| Docker sync events | Pouze log.Info | Audit log + alert |
| Config | Bez alerts sekce | `alerts:` sekce s webhook/slack/email |
| Event typy | 7 typů (SERVED, BLOCKED, ...) | +1 nový: `TAG_MUTATED` |

## Návrh řešení

### Architektura

```
WriteAuditLog(db, entry)         ← existující funkce (base.go:76)
    │
    ├── INSERT INTO audit_log     ← beze změny
    │
    └── alerter.Dispatch(entry)   ← NOVÉ: asynchronní dispatch
            │
            ├── filter: entry.EventType ∈ channel.On ?
            │
            ├── Webhook channel   → POST JSON na URL
            ├── Slack channel     → POST na Slack webhook URL
            └── Email channel     → SMTP odesílání
```

**Klíčová rozhodnutí:**

1. **Hook na WriteAuditLog, ne na policy engine** — alert se váže na auditovanou událost, ne na policy rozhodnutí. Jeden choke-point, žádné duplikace.
2. **Asynchronní dispatch** — alert NESMÍ blokovat request flow. Fire-and-forget s retry v pozadí.
3. **Filtrace per-kanál** — každý kanál má seznam event typů (`on: ["BLOCKED", "QUARANTINED"]`), na které reaguje.
4. **Nový event type `TAG_MUTATED`** — pro Docker sync digest change, doplní se do `model/audit.go`.

### Interface

```go
// internal/alert/alerter.go

// Alerter sends notifications for audit events.
type Alerter interface {
    // Dispatch sends an alert for the given audit entry.
    // Must be non-blocking (implementation should use goroutines/channels internally).
    Dispatch(entry model.AuditEntry)

    // Close gracefully shuts down the alerter, flushing pending alerts.
    Close() error
}
```

**MultiAlerter** — agreguje více kanálů, filtruje per-kanál na základě `On` seznamu event typů:

```go
type ChannelConfig struct {
    Channel Alerter
    On      []model.EventType  // prázdný = ALL events
}

type MultiAlerter struct {
    channels []ChannelConfig
    queue    chan model.AuditEntry  // buffered channel pro async dispatch
    wg       sync.WaitGroup
}
```

### Změny v `WriteAuditLog`

Centrální místo pro hook — `internal/adapter/base.go:76-97`:

```go
// PŘED:
func WriteAuditLog(db *sqlx.DB, entry model.AuditEntry) error { ... }

// PO:
func WriteAuditLog(db *sqlx.DB, alerter alert.Alerter, entry model.AuditEntry) error {
    // ... existující INSERT do audit_log (beze změny) ...

    // Dispatch alert (async, non-blocking)
    if alerter != nil {
        alerter.Dispatch(entry)
    }
    return nil
}
```

**Alternativa (preferovaná):** Aby se neměnila signatura `WriteAuditLog` na všech call-sites, uložit alerter jako package-level proměnnou:

```go
var globalAlerter alert.Alerter

func SetAlerter(a alert.Alerter) {
    globalAlerter = a
}

func WriteAuditLog(db *sqlx.DB, entry model.AuditEntry) error {
    // ... existující INSERT ...

    if globalAlerter != nil {
        globalAlerter.Dispatch(entry)
    }
    return nil
}
```

Toto je pragmatičtější — nulové změny na call-sites (pypi.go, npm.go, nuget.go, docker.go, artifacts.go, overrides.go).

## Rizika a mitigace

| Riziko | Dopad | Pravděpodobnost | Mitigace |
|--------|-------|-----------------|----------|
| Alert selhání zpomalí request flow | Vysoký — latence pro klienty | Nízká (async dispatch) | Dispatch je fire-and-forget přes buffered channel; selhání se loguje, neblokuje |
| Webhook endpoint nedostupný | Střední — ztráta alertů | Střední | Retry s exponential backoff (max 3 pokusy); audit_log zůstává jako source of truth |
| SMTP timeout (email) | Střední — goroutine leak | Střední | Timeout na SMTP dial (10s) a send (30s); buffered channel s limitem (1000) |
| Credentials leak v config souboru | Kritický | Nízká (design prevence) | Všechny secrets přes env var reference (`*_env` fields), NIKDY plaintext v YAML |
| Package-level globalAlerter | Nízký — testovatelnost | Nízká | SetAlerter/globalAlerter je pragmatický trade-off; v testech se nastaví mock |
| Změna signatury WriteAuditLog | Střední — breaking change | N/A (vyhneme se) | Použijeme package-level SetAlerter(), signatura WriteAuditLog se nemění |
| Buffer overflow při burst alertů | Střední — ztráta alertů | Nízká | Buffered channel (1000 cap); při plném bufferu se alert dropne a loguje warning |

## Poznámky

- **Idempotence:** Alerty jsou fire-and-forget, žádný dedup. Webhook endpointy by měly být idempotentní (audit_log.id lze použít jako idempotency key).
- **Řazení:** Alerty nejsou garantovaně v pořadí — buffered channel + goroutiny. Pro audit trail slouží tabulka `audit_log`.
- **Rate limiting:** V1.1 nemá built-in rate limiting na alerty. Pokud je burst eventů (např. threat feed update karanténuje 100 artefaktů), všechny se odešlou. Rate limiting je kandidát na v1.2.
- **Zpětná kompatibilita:** 100% zpětně kompatibilní — alerty jsou opt-in (disabled by default). Signatura WriteAuditLog se nemění.
- **Email security:** SMTP credentials MUSÍ být z env proměnných. Config soubor obsahuje pouze reference na env var jméno, nikdy samotné heslo.
- **Go závislost:** Žádná nová externí závislost potřeba. `net/smtp` je stdlib, `net/http` pro webhook/slack je stdlib. HMAC je `crypto/hmac` + `crypto/sha256` (stdlib).
