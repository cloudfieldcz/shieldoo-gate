# v1.1 Alerting System (Webhook / Slack / Email) — Technická analýza

## Popis

Implementace alertovacího systému pro Shieldoo Gate, který odesílá notifikace při bezpečnostních událostech (blokace, karanténa, změna digestu tagu, release z karantény). Systém podporuje tři kanály: generický webhook, Slack a email (SMTP).

### Proč

- Alerting je **prerekvizita** pro rescan scheduler (v1.1) a plnohodnotnou tag mutability detection — bez notifikací administrátor neví, že se něco stalo
- Současný stav: bezpečnostní události jsou pouze logovány (zerolog) a zapisovány do audit_log tabulky — žádný proaktivní push k operátorovi
- Docker sync service (`sync.go:242-251`) karanténuje artefakty bez jakékoli notifikace
- Analýza `initial-analyse.md:515-528` definuje alerting config, ale implementace chybí

### Out of Scope

- UI konfigurace alertů (separátní scope, budoucí práce)
- Hot-reload konfigurace alertů (vyžaduje restart)
- API endpoint pro test alertů ("send test alert")
- Alert delivery history / status tracking
- WARNED event alerting (může být přidán později, v1.0 nemá warn policy action v produkci)
- `TAG_MUTATED` event **generování** — tato analýza přidává pouze konstantu event typu pro forward-kompatibilitu. Samotná detekce tag mutability (config sekce `policy.tag_mutability` s `action`, `exclude_tags`, `check_on_cache_hit`, `check_interval` dle `initial-analyse.md:784-797`) je součástí separátní analýzy "Tag Mutability Detection". Docker sync (`sync.go:179`) již detekuje digest change a bude zapisovat `TAG_MUTATED` audit log po opravě v této analýze.
- Async/batch `WriteAuditLog` pro snížení SQLite write contention — viz Poznámky

## Aktuální stav

### Jak jsou události dnes zpracovány

Všechny bezpečnostní události procházejí dvěma cestami:
1. **Strukturované logy** (zerolog) — `log.Info()` / `log.Warn()` / `log.Error()`
2. **Audit log tabulka** — dvěma způsoby:
   - **Adaptéry** (pypi, npm, nuget, docker pull): `adapter.WriteAuditLog()` v `internal/adapter/base.go:76-97`
   - **API handlery** (artifacts, overrides): přímý SQL INSERT do `audit_log` v rámci transakce — **obchází `WriteAuditLog()`!**

Žádná třetí cesta (push notifikace) neexistuje.

### Architektonické poznatky

1. **WriteAuditLog je choke-point pouze pro adaptéry** — API handlery (`artifacts.go`, `overrides.go`) obcházejí `WriteAuditLog()` a zapisují přímo SQL v transakcích. Hook na `WriteAuditLog` nestačí — je nutné přidat alert dispatch i do API handlerů po commitu transakce.
2. **Docker sync nemá audit log** — sync.go volá `persistArtifact()` ale nikdy `WriteAuditLog()`. Toto je bug k opravě.
3. **Config nemá alerts sekci** — `internal/config/config.go` obsahuje pouze Server, Ports, Upstreams, Cache, Database, Scanners, Policy, ThreatFeed, Log.
4. **Adaptéry nemají přístup k alerteru** — konstruktory přijímají `(db, cache, scanEngine, policyEngine, upstream)`.

| Aspekt | Současný stav | Navrhovaný stav |
|--------|--------------|-----------------|
| Notifikace | Žádné | Webhook + Slack + Email |
| Audit log (adaptéry) | Zápis přes WriteAuditLog | Zápis + dispatch alertu |
| Audit log (API) | Přímý SQL v transakcích | Přímý SQL + explicitní dispatch po commitu |
| Docker sync events | Pouze log.Info | Audit log + alert |

## Návrh řešení

### Architektura

```
                          ┌──────────────────────────────────────┐
                          │          MultiAlerter                │
                          │                                      │
WriteAuditLog()           │  ┌─ WebhookWorker ──▶ HTTP POST     │
  └─▶ Dispatch(entry) ───▶│  │  (own goroutine, own queue)      │
                          │  │                                    │
API handlers              │  ├─ SlackWorker ────▶ Slack POST     │
  (after tx.Commit)       │  │  (own goroutine, own queue)      │
  └─▶ Dispatch(entry) ───▶│  │                                    │
                          │  └─ EmailWorker ────▶ SMTP           │
                          │     (own goroutine, own queue,       │
                          │      batching: digest per 30s)       │
                          └──────────────────────────────────────┘
```

**Klíčová rozhodnutí:**

1. **Hook na WriteAuditLog + explicitní dispatch v API handlerech** — adaptéry volají WriteAuditLog (automatický hook), API handlery volají `alerter.Dispatch()` explicitně po commitu transakce. Pokrývá všech 13 trigger pointů.
2. **Per-channel goroutine s nezávislým bufferem** — každý kanál má vlastní goroutinu a buffered channel. Pomalý SMTP neblokuje webhook/Slack. Žádný head-of-line blocking.
3. **Filtrace per-kanál** — každý kanál má seznam event typů (`on: ["BLOCKED", "QUARANTINED"]`). Prázdný `on` = ALL events. Podporuje i `"ALL"` jako explicitní hodnotu (kompatibilita s `initial-analyse.md`).
4. **Nové event typy** — `TAG_MUTATED` pro Docker sync digest change, `RESCAN_QUEUED` (konstanta pro existující string literal).
5. **Token-bucket rate limiting per kanál** — ochrana proti burst (threat feed update, 100+ karantén). Slack respektuje 1 msg/sec limit.
6. **Email batching** — emaily se kumulují po konfigurovatelný interval (default 30s) a odesílají jako jeden digest.

### Interface

```go
// internal/alert/alerter.go

// Alerter sends notifications for audit events.
type Alerter interface {
    // Dispatch sends an alert for the given audit entry.
    // Must be non-blocking (channel send internally).
    Dispatch(ctx context.Context, entry model.AuditEntry)

    // Close gracefully shuts down the alerter, flushing pending alerts.
    // Respects the provided timeout from context.
    Close() error
}
```

### AlertPayload — bezpečný subset pro odchozí komunikaci

```go
// internal/alert/payload.go

// AlertPayload je struktura odesílaná na webhook/Slack/email.
// NIKDY neobsahuje ClientIP, UserAgent ani raw MetadataJSON z AuditEntry.
type AlertPayload struct {
    EventType  string    `json:"event_type"`
    ArtifactID string    `json:"artifact_id"`
    Reason     string    `json:"reason"`
    Timestamp  time.Time `json:"timestamp"`
    // Strukturovaná metadata per event type — viz níže.
    Metadata   any       `json:"metadata,omitempty"`
}

// Metadata per event type:
// TAG_MUTATED:       {"old_digest": "sha256:...", "new_digest": "sha256:...", "ecosystem": "docker"}
// BLOCKED:           {"verdict": "MALICIOUS", "findings_count": 3}
// QUARANTINED:       {"verdict": "SUSPICIOUS", "findings_count": 1}
// RELEASED:          {} (žádná extra metadata)
// OVERRIDE_CREATED:  {"scope": "version", "expires_at": "..."}
```

**Bezpečnostní invariant:** `ClientIP`, `UserAgent` a raw `MetadataJSON` se NIKDY neodesílají na externí endpointy. Prevence úniku PII a interní síťové topologie.

### MultiAlerter — per-channel worker architektura

```go
type ChannelWorker struct {
    channel  ChannelSender           // Webhook/Slack/Email sender
    on       map[model.EventType]bool // filter; nil = ALL
    queue    chan AlertPayload        // per-channel buffer (default 256)
    limiter  *rate.Limiter           // token-bucket rate limiter
    wg       sync.WaitGroup
    cancel   context.CancelFunc
}

type MultiAlerter struct {
    workers []*ChannelWorker
    cancel  context.CancelFunc  // master cancel pro shutdown
}
```

Každý `ChannelWorker`:
- Má vlastní goroutinu čtoucí z `queue`
- Má vlastní `rate.Limiter` (webhook: 10/s, Slack: 1/s, email: N/A — batching)
- Při plném bufferu: drop + increment Prometheus counter `shieldoo_alerts_dropped_total{channel="..."}`
- Retry: max 3 pokusy s exponential backoff (1s, 2s, 4s), žádný retry na 4xx
- Circuit breaker: po 5 po sobě jdoucích selháních → cooldown 60s → zkusí znovu

### Webhook kanál

```go
// internal/alert/webhook.go

type WebhookSender struct {
    url        string
    httpClient *http.Client  // Timeout: 10s
    secretKey  []byte
}
```

Payload (JSON):
```json
{
  "event_type": "QUARANTINED",
  "artifact_id": "pypi:litellm:1.82.7",
  "reason": "verdict SUSPICIOUS meets quarantine threshold",
  "timestamp": "2026-03-28T14:30:00Z",
  "metadata": {"verdict": "SUSPICIOUS", "findings_count": 1}
}
```

HTTP hlavičky:
- `Content-Type: application/json`
- `X-Shieldoo-Event: QUARANTINED`
- `X-Shieldoo-Timestamp: 1711634400` (Unix timestamp, součást HMAC)
- `X-Shieldoo-Signature: sha256=<hex(HMAC-SHA256(secret, timestamp + "." + body))>`
- `User-Agent: shieldoo-gate/1.1`

**HMAC podpis:** `HMAC-SHA256(secret, "<timestamp>.<raw_json_body>")`. Timestamp v podpisu brání replay útokům. Příjemce by měl odmítnout zprávy starší než 5 minut.

**URL validace (SSRF ochrana):**
- Config validace vyžaduje `https://` schéma (pro HTTP je nutný explicitní `allow_insecure: true`)
- Blokace RFC 1918 / RFC 6598 / link-local / loopback adres (resolve hostname → check IP)
- Validace probíhá při startu v `Config.Validate()`

### Email kanál

```go
// internal/alert/email.go

type EmailSender struct {
    host      string
    port      int
    from      string
    to        []string
    username  string
    password  string   // z env proměnné, NIKDY z config souboru
    useTLS    bool
    batchWait time.Duration  // default 30s — kumuluje eventy do jednoho digest emailu
}
```

**Batching:** EmailWorker kumuluje eventy po `batchWait` interval. Po uplynutí odešle jeden email se seznamem všech eventů. Předchází zahlcení SMTP serveru při burst (100 karantén → 1 email s tabulkou, ne 100 emailů).

**Subject:** `[Shieldoo Gate] {N} security event(s): {EVENT_TYPES}`
- Příklad: `[Shieldoo Gate] 3 security event(s): QUARANTINED, BLOCKED`

**Body:** Plain text se strukturovanou tabulkou eventů.

**TLS:**
- Port 587: STARTTLS s ověřením certifikátu (`tls.Config{ServerName: host}`)
- Port 465: Implicit TLS
- `tls_skip_verify: true` k dispozici pro self-signed certs (default false, dokumentováno jako insecure)

**Retry:** 1 pokus (SMTP je stateful, retry je problematický). Selhání se loguje + Prometheus counter.

```go
    To            []string `mapstructure:"to"`
    UsernameEnv   string   `mapstructure:"username_env"`
    PasswordEnv   string   `mapstructure:"password_env"`
    UseTLS        bool     `mapstructure:"use_tls"`
    TLSSkipVerify bool     `mapstructure:"tls_skip_verify"` // default false
    BatchInterval string   `mapstructure:"batch_interval"`  // default "30s"
    On            []string `mapstructure:"on"`

```

**Validační pravidla (`Config.Validate()`):**
- `webhook.enabled && url == ""` → startup error
- `webhook.enabled && !allow_insecure && !strings.HasPrefix(url, "https://")` → startup error
- `webhook.enabled && url resolves to RFC1918/loopback` → startup error
- `slack.enabled && webhook_env == ""` → startup error
- `slack.enabled && os.Getenv(webhook_env) == ""` → startup warning (env var not set)
- `slack.enabled && !strings.HasPrefix(resolved_url, "https://")` → startup error
- `email.enabled && (host == "" || from == "" || len(to) == 0)` → startup error
- `email.enabled && password_env != "" && os.Getenv(password_env) == ""` → startup warning
- `*.on` values validovány proti známým `EventType` konstantám → warning pro neznámé (forward-compatible)

### Změny v `WriteAuditLog` — package-level alerter

`internal/adapter/base.go:76-97`:

```go
// Thread-safe global alerter. Set once at startup before HTTP servers start.
var globalAlerter atomic.Pointer[alert.Alerter]

func SetAlerter(a alert.Alerter) {
    globalAlerter.Store(&a)
}

func WriteAuditLog(db *sqlx.DB, entry model.AuditEntry) error {
    // ... existující INSERT do audit_log (beze změny) ...

    // Dispatch alert (async, non-blocking)
    if a := globalAlerter.Load(); a != nil {
        (*a).Dispatch(context.Background(), entry)
    }
    return nil
}
```

`atomic.Pointer` zajišťuje thread-safety bez mutexu. Signatura `WriteAuditLog` se NEMĚNÍ — nulové změny na call-sites adaptérů.

### Změny v API handlerech — explicitní dispatch po commitu

API handlery (`artifacts.go`, `overrides.go`) zapisují audit log přímým SQL v transakcích. Nelze je hookovat přes `WriteAuditLog`. Řešení: po úspěšném commitu transakce zavolat `adapter.DispatchAlert(entry)`:

```go
// internal/adapter/base.go — nová helper funkce

// DispatchAlert sends an alert for the given entry without writing to audit_log.
// Use this when audit_log was already written (e.g., inside a transaction).
func DispatchAlert(entry model.AuditEntry) {
    if a := globalAlerter.Load(); a != nil {
        (*a).Dispatch(context.Background(), entry)
    }
}
```

Použití v `artifacts.go` (po commitu transakce):
```go
if err := tx.Commit(); err != nil { ... }
adapter.DispatchAlert(model.AuditEntry{
    EventType:  model.EventQuarantined,
    ArtifactID: id,
    Reason:     reason,
})
```

Stejný pattern pro: `handleReleaseArtifact`, `handleRescanArtifact`, `handleCreateOverride`, `handleRevokeOverride`.

### Shutdown sekvence v main.go

Pořadí při graceful shutdown:
1. Stop accepting new HTTP connections
2. Drain in-flight HTTP requests (stávající `srv.Shutdown(15s)`)
3. **`alerter.Close()` s timeoutem 5s** — flush pending alerts z per-channel bufferů
4. `db.Close()` — uzavřít databázi

```go
// main.go — po inicializaci alerteru:
defer func() {
    closeCtx, closeCancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer closeCancel()
    _ = alerter.Close() // flush pending, respektuje interní cancel
}()
```

## Akceptační kritéria

1. Když je webhook enabled s `on: ["BLOCKED"]`, PyPI BLOCKED event triggeruje přesně jeden HTTP POST na konfigurovanou URL do 5 sekund.
2. Když jsou všechny kanály disabled, žádné alerty se neodesílají a žádné errory se nelogují.
3. Když webhook endpoint vrátí HTTP 500, systém retryuje max 3x s exponential backoff, pak loguje warning a dropne alert.
4. Když webhook endpoint vrátí HTTP 400/401, žádný retry.
5. Signatura `WriteAuditLog` zůstává nezměněná; všechny existující call-sites kompilují bez modifikace.
6. Manual quarantine přes API (`POST /api/v1/artifacts/{id}/quarantine`) odešle alert na konfigurované kanály.
7. Docker sync digest change generuje `TAG_MUTATED` audit log záznam a alert.
8. Při burst 100 QUARANTINED eventů: webhook obdrží všech 100, Slack obdrží max 1/sec, email odešle 1 digest email.
9. Alert payload NIKDY neobsahuje `client_ip` ani `user_agent`.
10. Webhook URL s HTTP schématem bez `allow_insecure: true` způsobí startup error.

## Rizika a mitigace

| Riziko | Dopad | Pravděpodobnost | Mitigace |
|--------|-------|-----------------|----------|
| Alert selhání zpomalí request flow | Vysoký — latence pro klienty | Nízká | Dispatch je non-blocking channel send; per-channel worker goroutiny |
| Webhook endpoint nedostupný | Střední — ztráta alertů | Střední | Retry 3x exponential backoff + circuit breaker (5 failures → 60s cooldown); audit_log je source of truth |
| SMTP timeout blokuje jiné kanály | Vysoký — head-of-line blocking | N/A (vyřešeno) | Per-channel goroutiny — email worker je izolovaný od webhook/Slack |
| Goroutine leak při retry | Střední — OOM | Nízká | Fixed worker (1 goroutine per channel), retry v rámci workeru, context cancellation při Close() |
| Credentials leak v config | Kritický — security breach | Nízká (design) | Secrets přes env var reference (`*_env`), NIKDY plaintext |
| SSRF přes webhook URL | Vysoký — internal network exposure | Střední | HTTPS enforced, RFC1918/loopback blokace, validace při startu |
| PII leak v alert payload | Střední — GDPR violation | N/A (vyřešeno) | AlertPayload struct bez ClientIP/UserAgent/MetadataJSON |
| Burst alertů (threat feed update) | Střední — zahlcení kanálů | Střední | Token-bucket rate limiter per channel; email batching; Prometheus counter pro dropped alerts |
| HMAC replay útok | Nízký | Nízká | Timestamp v HMAC podpisu; příjemce odmítá zprávy starší 5 minut |
| Buffer overflow (per-channel 256) | Střední — ztráta alertů | Nízká | Drop + Prometheus counter + log warning s event_type a artifact_id; audit_log je definitivní záznam |
| `atomic.Pointer` globalAlerter | Nízký — code complexity | N/A | stdlib `sync/atomic`; jednoduchý pattern, thread-safe |

## Poznámky

- **Idempotence:** Alerty jsou fire-and-forget, žádný dedup. Webhook endpointy by měly být idempotentní (`audit_log.id` v metadata jako idempotency key).
- **Řazení:** Alerty nejsou garantovaně v pořadí — per-channel goroutiny s nezávislými buffery. Pro audit trail slouží tabulka `audit_log`.
- **Zpětná kompatibilita:** 100% zpětně kompatibilní — alerty jsou opt-in (disabled by default). Signatura WriteAuditLog se nemění.
- **Email security:** SMTP credentials MUSÍ být z env proměnných. TLS s ověřením certifikátu je default; `tls_skip_verify` je opt-in a dokumentován jako insecure.
- **Go závislosti:** Jedna nová závislost: `golang.org/x/time/rate` pro token-bucket rate limiter. Ostatní stdlib (`net/smtp`, `net/http`, `crypto/hmac`, `crypto/sha256`, `sync/atomic`).
- **SQLite write contention (pre-existing):** `WriteAuditLog` provádí synchronní INSERT v request path. Přidání `Dispatch()` (non-blocking channel send) toto nezhoršuje, ale samotný INSERT je bottleneck při 50+ concurrent requests. Řešení (async batch INSERT do audit_log) je mimo scope této analýzy — kandidát na separátní optimalizaci. Alerting systém na tomto nezávisí (čte z AuditEntry struct, ne z DB).
- **Rescan scheduler kompatibilita:** Budoucí rescan scheduler (v1.1) automaticky benefituje z alert hooku — jeho volání `WriteAuditLog` spustí alerty bez dalšího wiring.
- **Viper AutomaticEnv interakce:** Config polí `*_env` jsou jména env proměnných, nikoli samotné hodnoty. Viper s prefixem `SGW_` může overridovat config klíče (`SGW_ALERTS_WEBHOOK_URL`), ale `*_env` pole se čtou přes `os.Getenv()`, nikoli přes viper. Tato interakce je zdokumentována v `docs/configuration.md`.
