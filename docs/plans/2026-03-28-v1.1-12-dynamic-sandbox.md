# v1.1 Dynamic Sandbox Execution (gVisor) — Technická analýza

## Popis

Přidání runtime sandbox pro skenování artefaktů. Místo statické analýzy (aktuální stav) se artefakt spustí/nainstaluje v izolovaném gVisor kontejneru a monitoruje se jeho chování (síťové hovory, filesystem přístupy, process execution).

### Proč

- Statická analýza má blind spots: dostatečně sofistikovaná obfuskace (double base64, encrypted payloads) projde
- LiteLLM útok použil `.pth` soubor — execute-on-import, detekovaný staticky, ale budoucí varianty mohou být sofistikovanější
- `initial-analyse.md:1049` uvádí "dynamic sandbox deferred to v1.1"
- gVisor (runsc) poskytuje syscall-level izolaci bez full VM overhead

### Out of Scope

- Full malware analysis platform — pouze install-time behavior monitoring
- Windows containers — pouze Linux
- Custom sandbox (non-gVisor) — pouze gVisor/runsc

## Aktuální stav

### Scanner execution

Trivy: `exec.CommandContext(ctx, "trivy", ...)` — subprocess na hostu (`internal/scanner/trivy/trivy.go:48-82`)
- Žádná izolace, přímý přístup k host filesystem
- Timeout: 30s default

GuardDog: gRPC přes Unix socket (`internal/scanner/guarddog/guarddog.go:54-87`)
- Scanner bridge může být kontejnerizovaný (docker-compose sidecar)
- Ale samotný scan je statická analýza

Built-in scanners: Pure Go, in-process (`internal/scanner/builtin/`)
- Čtou artefakt z `artifact.LocalPath`
- Žádné spouštění artefaktu

### Artifact passing

Všechny scanners přijímají `artifact.LocalPath` — cestu k souboru na lokálním filesystému. Dynamic sandbox bude potřebovat:
1. Zkopírovat artefakt do sandboxu
2. Spustit install command (`pip install`, `npm install`)
3. Monitorovat chování
4. Vrátit výsledky

| Aspekt | Současný stav | Navrhovaný stav |
|--------|--------------|-----------------|
| Analýza | Statická pouze | Statická + dynamická |
| Izolace | Žádná (Trivy na hostu) | gVisor sandbox |
| Detekce | Pattern matching | + behavioral analysis |
| Overhead | Nízký (~1-5s) | Vyšší (~10-30s per artifact) |

## Návrh řešení

### Architektura

```
Scanner Engine
    │
    ├── Built-in scanners (statická analýza, beze změny)
    ├── GuardDog (gRPC bridge, beze změny)
    ├── Trivy (subprocess, beze změny)
    ├── OSV (API, beze změny)
    │
    └── SandboxScanner (NOVÝ)
            │
            ├── 1. Vytvořit gVisor kontejner (runsc)
            ├── 2. Zkopírovat artefakt dovnitř
            ├── 3. Spustit ecosystem-specific install command
            ├── 4. Monitorovat:
            │       ├── Network calls (DNS, HTTP)
            │       ├── File system writes (outside expected paths)
            │       ├── Process execution (suspicious binaries)
            │       └── Syscall patterns
            ├── 5. Timeout (30s default)
            └── 6. Vrátit ScanResult s findings
```

### SandboxScanner

```go
// internal/scanner/sandbox/sandbox.go

type SandboxScanner struct {
    runtimeBinary string    // path to "runsc" binary
    baseImage     map[scanner.Ecosystem]string  // per-ecosystem base images
    timeout       time.Duration
    networkPolicy string    // "none" | "monitor" — sandbox network mode
}

func (s *SandboxScanner) Name() string { return "sandbox" }
func (s *SandboxScanner) SupportedEcosystems() []scanner.Ecosystem {
    return []scanner.Ecosystem{
        scanner.EcosystemPyPI,
        scanner.EcosystemNPM,
        // Postupně další ekosystémy
    }
}
```

### Per-ecosystem install commands

| Ecosystem | Base Image | Install Command | Monitor |
|-----------|-----------|-----------------|---------|
| PyPI | `python:3.12-slim` | `pip install --no-deps <artifact.whl>` | DNS, HTTP, filesystem |
| npm | `node:20-slim` | `npm install <artifact.tgz>` | DNS, HTTP, postinstall |

| Maven | `maven:3-eclipse-temurin-21` | `mvn dependency:copy` | DNS, HTTP, init scripts |

### gVisor (runsc) integration

```go
func (s *SandboxScanner) runInSandbox(ctx context.Context, artifact scanner.Artifact) ([]Finding, error) {
    // 1. Prepare rootfs with base image
    // 2. Copy artifact to rootfs
    // 3. Create OCI runtime spec with:
    //    - Network: none (or monitored bridge)
    //    - Filesystem: read-only except /tmp
    //    - Syscall filter: default gVisor policy
    //    - Resource limits: CPU 1 core, Memory 512MB
    // 4. Run via "runsc run" with timeout
    // 5. Collect logs: DNS queries, HTTP connections, file writes
    // 6. Analyze logs for suspicious behavior
    // 7. Return findings
}
```

### Behavioral indicators

| Indicator | Severity | Description |
|-----------|----------|-------------|
| DNS query to non-registry domain | HIGH | Package resolves unknown domain during install |
| HTTP POST to external URL | CRITICAL | Data exfiltration attempt |
| Write to `~/.ssh/`, `~/.config/` | CRITICAL | Credential theft (LiteLLM pattern) |
| Execute `/bin/sh`, `/bin/bash` | HIGH | Shell execution during install |
| Read environment variables | MEDIUM | Possible credential harvesting |
| Create cron job | CRITICAL | Persistence mechanism |
| Write `.pth` file | CRITICAL | Python auto-execute (LiteLLM exact vector) |

### Konfigurace

```go
type SandboxConfig struct {
    Enabled        bool   `mapstructure:"enabled"`
    RuntimeBinary  string `mapstructure:"runtime_binary"`  // default "runsc"
    Timeout        string `mapstructure:"timeout"`          // default "30s"
    NetworkPolicy  string `mapstructure:"network_policy"`   // "none" | "monitor"

```

```yaml
scanners:

    enabled: false

    timeout: "30s"

    max_concurrent: 2
```

### Scanner interface compliance

```go
var _ scanner.Scanner = (*SandboxScanner)(nil)
```

SandboxScanner implementuje `scanner.Scanner` interface identicky jako ostatní scanery. Scanner Engine ho automaticky spustí paralelně s ostatními.

## Rizika a mitigace

| Riziko | Dopad | Pravděpodobnost | Mitigace |
|--------|-------|-----------------|----------|
| gVisor nedostupný na platformě | Vysoký — scanner nefunkční | Střední | `enabled: false` default; optional scanner, fail-open |
| Vysoká latence (10-30s per scan) | Střední — pomalé první stažení | Vysoká | `max_concurrent: 2`; parallel s ostatními scanery |
| Container escape | Kritický | Velmi nízká | gVisor syscall interception; read-only rootfs; no capabilities |
| False positives (legitimní network calls) | Střední — blokace validních balíků | Střední | Allowlist pro known registries; `warn` mode; confidence thresholds |
| Resource exhaustion (fork bomb v sandboxu) | Střední | Nízká | gVisor PID limits; cgroup memory/CPU limits |

## Poznámky

- **Toto je nejkomplexnější feature v1.1.** Doporučuji implementovat jako poslední, po stabilizaci ostatních features.
- **gVisor requirement:** Vyžaduje Linux host s gVisor/runsc nainstalovaným. Nelze použít na macOS/Windows (development fallback: Docker s `--runtime=runsc`).
- **Fail-open:** Pokud sandbox scan selže (runsc chybí, timeout, container error), vrátí VerdictClean. Konzistentní s celkovým design principem.
