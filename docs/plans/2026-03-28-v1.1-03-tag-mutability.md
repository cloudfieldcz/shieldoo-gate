# v1.1 Tag Mutability Detection (all ecosystems) — Technická analýza

## Popis

Rozšíření tag mutability detection ze současné Docker-only implementace na všechny ekosystémy (PyPI, npm, NuGet). Detekce, kdy upstream tag/verze ukazuje na jiný digest než dříve cachovaný artefakt — signál supply chain útoku.

### Proč

- Supply chain útok LiteLLM: útočník přepsal existující verzi na PyPI novým obsahem
- Docker sync (`sync.go:179`) již detekuje digest change pro Docker tagy — ale PyPI/npm/NuGet nemají ekvivalentní mechanismus
- `initial-analyse.md:733-797` definuje kompletní design včetně DB tabulky `tag_digest_history` a konfigurace
- Bez detekce na non-Docker ekosystémech: útočník přepíše PyPI balík, Shieldoo Gate serví z cache starý čistý artefakt, ale při cache expiraci stáhne nový malicious artefakt bez varování

### Out of Scope

- Docker tag mutability — již implementováno v `sync.go`, tato analýza ho pouze integruje s `tag_digest_history` tabulkou
- UI pro tag mutability eventy
- `check_interval` periodic background checks (fáze 2 — vyžaduje rescan scheduler integraci)

### Společný pattern stahování

Všechny 3 adaptéry mají identický download pattern:
1. `downloadToTemp()` → `(tmpPath, size, sha256hex, error)`
2. Build `scanner.Artifact` s SHA256
3. `InsertArtifact()` → DB s `sha256` sloupcem

**Klíčový poznatek:** SHA256 artefaktu je **již uložen v `artifacts` tabulce** (`artifacts.sha256`, migration 001). Pro detekci stačí při cache hit porovnat upstream SHA256 s uloženým.

| Aspekt | Současný stav | Navrhovaný stav |
|--------|--------------|-----------------|
| Docker digest check | Ano (sync.go) | Ano + tag_digest_history |
| PyPI digest check | Ne | Ano (na cache hit) |
| npm digest check | Ne | Ano (na cache hit) |
| NuGet digest check | Ne | Ano (na cache hit) |
| tag_digest_history | Neexistuje | Nová tabulka pro historii |
| TAG_MUTATED event | Definován (v1.1-01) | Zapisován při detekci |

## Návrh řešení

### Architektura

```
Client request → Adapter cache hit path
    │
    ├── Artifact in cache with CLEAN status?
    │       └── YES → check_on_cache_hit enabled?
    │                   │
    │                   ├── YES → HEAD request na upstream
    │                   │          ├── SHA256 matches → serve from cache
    │                   │          └── SHA256 CHANGED → TAG_MUTATED flow
    │                   │
    │                   └── NO → serve from cache (bez check)
    │
    └── NO → normal download flow (stávající)
```

**TAG_MUTATED flow:**
1. Quarantine NOVÝ artefakt (nestahovat, neservit)
2. Starý cachovaný artefakt zůstává CLEAN (servíruje se dál)
3. Zapsat do `tag_digest_history`
4. WriteAuditLog s `EventTagMutated`
5. Alert se triggeruje automaticky (v1.1-01)

### Detekce upstream digestu bez full download

**PyPI:** `HEAD` request na package URL → `Content-Length` + `ETag` header. PyPI vrací `ETag` s SHA256 hashem. Pokud `ETag` chybí, fallback na `Content-Length` comparison (méně spolehlivé) nebo skip check.

**npm:** Registry API vrací `dist.shasum` (SHA1) a `dist.integrity` (SHA512 SRI) v metadata JSON. Porovnání s uloženým SHA256 vyžaduje buď: (a) uložit i shasum/integrity při prvním stažení, nebo (b) stáhnout metadata JSON a porovnat `dist.integrity`.

**NuGet:** V3 API vrací `catalogEntry.packageHash` (SHA512) a `packageHashAlgorithm`. Podobně jako npm — uložit hash při prvním stažení.

**Pragmatický přístup:** Pro v1.1 použít **lightweight HEAD/metadata check** kde je k dispozici, s fallbackem na "skip check" pro ekosystémy kde upstream neposkytuje stabilní hash v metadatech. Full download comparison je příliš drahý pro cache hits.

```go

// the artifact content has changed since it was cached.
// Returns (changed bool, newDigest string, err error).
// On error, returns (false, "", err) — fail-open, do not block serving.
```

### Nová tabulka `tag_digest_history`

```sql
CREATE TABLE IF NOT EXISTS tag_digest_history (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    ecosystem       TEXT NOT NULL,
    name            TEXT NOT NULL,
    tag_or_version  TEXT NOT NULL,
    digest_sha256   TEXT NOT NULL,
    first_seen_at   DATETIME NOT NULL,
    last_seen_at    DATETIME NOT NULL,
    superseded_at   DATETIME,
    UNIQUE(ecosystem, name, tag_or_version, digest_sha256)
);

CREATE INDEX IF NOT EXISTS idx_tag_digest_lookup
    ON tag_digest_history(ecosystem, name, tag_or_version);
```

### Konfigurace

Nová sekce v `PolicyConfig`:

```go
type TagMutabilityConfig struct {
    Enabled         bool     `mapstructure:"enabled"`
    Action          string   `mapstructure:"action"`            // quarantine | block | warn
    ExcludeTags     []string `mapstructure:"exclude_tags"`      // ["latest", "nightly", "dev"]
    CheckOnCacheHit bool     `mapstructure:"check_on_cache_hit"` // true = HEAD check na cache hit
}
```

YAML:
```yaml
policy:
  tag_mutability:
    enabled: true
    action: quarantine
    exclude_tags: ["latest", "nightly", "dev"]
    check_on_cache_hit: true
```

### Změny v adaptérech — cache hit path

Každý adaptér (pypi, npm, nuget) má cache hit path kde serví z cache. Sem se přidá digest check:

```go
// Pseudokód — cache hit path v každém adaptéru
if cachedStatus.IsServable() {
    // NOVÉ: tag mutability check (pokud enabled)
    if cfg.TagMutability.Enabled && cfg.TagMutability.CheckOnCacheHit {
        if !isExcludedTag(version, cfg.TagMutability.ExcludeTags) {
            changed, newDigest, err := CheckDigestChanged(ctx, ecosystem, upstreamURL, artifact.SHA256)
            if err == nil && changed {
                // TAG_MUTATED flow
                recordDigestHistory(db, ecosystem, name, version, artifact.SHA256, newDigest)
                writeAuditLog(EventTagMutated, metadata: {old: artifact.SHA256, new: newDigest})
                // Action based on config
                switch cfg.TagMutability.Action {
                case "quarantine": // quarantine new, serve old
                case "block":     // block request entirely
                case "warn":      // serve old, log warning
                }
            }
            // On error: fail-open, serve from cache normally
        }
    }
    // serve from cache
}
```

### Latence concern

`check_on_cache_hit: true` přidává HEAD request na upstream při každém cache hit. Pro PyPI/npm/NuGet to je ~50-200ms latence. Operátor může vypnout (`check_on_cache_hit: false`) a spoléhat na:
- Rescan scheduler (v1.1-02) pro periodickou kontrolu
- `check_interval` background checks (budoucí fáze)

## Akceptační kritéria

1. Při cache hit na PyPI artefakt s `check_on_cache_hit: true`: HEAD request ověří upstream digest.
2. Pokud se digest změnil: audit log `TAG_MUTATED` + alert + action dle config.
3. Tagy v `exclude_tags` se nekontrolují.
4. Při `action: quarantine`: starý artefakt se stále servíruje, nový se neservíruje.
5. Při `action: warn`: artefakt se servíruje, loguje se warning.
6. HEAD request failure → fail-open, serví z cache.
7. `tag_digest_history` tabulka obsahuje historii všech digestů pro každý tag/verzi.
8. Docker sync zapisuje do `tag_digest_history` při digest change.

## Rizika a mitigace

| Riziko | Dopad | Pravděpodobnost | Mitigace |
|--------|-------|-----------------|----------|
| Latence na cache hit (HEAD request) | Střední — +50-200ms per request | Vysoká | `check_on_cache_hit` je opt-in; fallback na rescan scheduler |
| Upstream nemá stabilní hash v metadatech | Střední — false negatives | Střední | Fail-open; dokumentovat omezení per ekosystém |
| False positive (legitimní verze update) | Vysoký — blokace validního artefaktu | Nízká | `exclude_tags` pro známé mutable tagy; `action: warn` jako konzervativní default |
| Race condition cache hit vs eviction | Nízký | Nízká | Graceful handling: fail-open |

## Poznámky

- **Vztah k v1.1-01 (Alerts):** `TAG_MUTATED` event type je definován v alerts analýze. Alert se triggeruje automaticky přes WriteAuditLog hook.
- **Vztah k v1.1-02 (Rescan Scheduler):** Rescan scheduler může sloužit jako alternativa k `check_on_cache_hit` — periodicky ověřuje digesty cachovaných artefaktů.
- **npm specifika:** npm umožňuje republish do 72h. `dist.integrity` (SHA512 SRI) je nejspolehlivější identifikátor.
- **PyPI specifika:** PyPI po smazání verze neumožňuje re-upload stejné verze. Ale kompromitovaný maintainer účet může nahrát novou verzi.
