## E2E testovací strategie — KRITICKÝ CROSS-CUTTING CONCERN

### Proč je E2E testování klíčové pro v1.1

v1.1 přináší **zásadní změny v každé vrstvě systému** — nové storage backendy, nové databázové backendy, nové protokolové adaptéry, autentizaci, scheduling, sandbox execution. Každá z těchto změn může subtilně rozbít stávající fungující flow. **Bez kvalitních E2E testů pro každou feature nemáme žádnou jistotu, že celek funguje.**

Nestačí unit testy — ty ověřují izolované komponenty. E2E testy ověřují **celý pipeline** od klientského requestu (pip install, docker pull) přes proxy → scan → policy → cache → storage → DB → alert. Pouze E2E test odhalí:
- Rozbité wire-up v main.go
- Nekompatibilní SQL mezi SQLite a PostgreSQL
- Chybějící cleanup temp souborů v cloud storage flow
- Race conditions mezi rescan schedulrem a live requesty
- Auth middleware blokující proxy endpointy (nesmí)

### Aktuální stav E2E testů

- Shell-based testy v `tests/e2e-shell/` — pokrývají PyPI, npm, NuGet, Docker (základní happy path)
- Go E2E testy v `tests/e2e/e2e_test.go` — build tag `//go:build e2e`
- Test klienti běží na hostu (ne kontejnerizovaní) — řeší v1.1-11
- **Gap:** Žádné E2E testy pro alerting, rescan, tag mutability, auth, cloud storage, PostgreSQL

### E2E test architektura pro v1.1

```

    │
    ├── shieldoo-gate           (main binary)
    ├── scanner-bridge          (Python sidecar)
    ├── push-registry           (Docker push target)
    │
    ├── postgresql              (NOVÉ — PG backend pro dual-backend testy)
    ├── minio                   (NOVÉ — S3-compatible storage pro cloud testy)
    ├── webhook-receiver        (NOVÉ — HTTP server pro alert verifikaci)
    ├── mock-oidc-provider      (NOVÉ — fake OIDC provider pro auth testy)
    │

        ├── dotnet

        ├── go
        └── test scripts + assertions
```

### Zásady pro E2E testy v1.1

1. **Každá feature MUSÍ mít alespoň jeden E2E test** — nestačí unit testy, musí projít celý flow od klienta po storage/DB.
2. **E2E test je součást feature delivery** — implementuje se SPOLEČNĚ s feature, NE jako follow-up. Každá analýza má E2E test v implementační fázi.
3. **Dual-backend CI** — E2E test suite běží 2x: jednou s SQLite + local cache, jednou s PostgreSQL + MinIO. Obě konfigurace musí projít.
4. **Alert verification** — webhook-receiver (jednoduchý HTTP server logující requesty) ověřuje, že alerty dorazily se správným payload.
5. **Malicious test packages** — testovací "malicious" balíky (s neškodným ale detekovatelným patternem, např. base64-encoded `echo test`) pro ověření block/quarantine flow.
6. **Regression safety** — všechny stávající v1.0 E2E testy musí projít i po v1.1 změnách. Žádná v1.1 feature nesmí rozbít v1.0 functionality.
7. **Kontejnerizace** — zero host dependencies (řeší v1.1-11). CI pipeline buildí test-runner image a spouští testy uvnitř.
8. **Reproducibilita** — deterministic výsledky. Mock upstreamy kde možno (mock PyPI, mock npm registry).
9. **Celkový smoke test** — end-to-end test, který spustí celý v1.1 stack (PG + S3 + auth + alerts + rescan) a provede kompletní workflow: stáhni balík → skenuj → servíruj → rescan → quarantine → alert → verify webhook. Toto je finální "works together" verifikace.
