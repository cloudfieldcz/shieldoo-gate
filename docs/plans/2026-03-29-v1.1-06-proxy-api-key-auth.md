# Proxy API Key Authentication — Autentikace proxy endpointů pomocí API klíčů

## Popis

Přidání autentikace na proxy endpointy (PyPI, npm, NuGet, Docker, Maven, RubyGems, Go Modules) pomocí API klíčů přenášených přes HTTP Basic Auth.

### Proč

- Aktuálně jsou proxy endpointy zcela otevřené — ochrana spoléhá pouze na síťovou segmentaci
- V praxi není síťová segmentace vždy dostatečná (shared VPC, VPN přístup, cloud prostředí)
- Potřeba auditovat KDO stahuje balíčky, nejen CO se stahuje
- Standardní package managery (pip, npm, docker, nuget, gem, go) nativně podporují HTTP Basic Auth

### Tři režimy

| Režim | `proxy_auth.enabled` | Chování |
|-------|----------------------|---------|
| **Off** (default) | `false` | Bez autentikace — zpětná kompatibilita |
| **On — PAT** | `true` | Per-user Personal Access Token generovaný přes admin API |
| **On — globální token** | `true` + `global_token_env` | Centrální token z config/env — pro prostředí bez OIDC |

Oba typy tokenů (PAT i globální) jsou akceptovány současně. Globální token je fallback pro případy, kdy není zapnutá OIDC autentikace admin API (a tedy nelze generovat PAT).

## Návrh řešení

### Architektura

```
pip install --index-url https://user:TOKEN@gate:5010/simple/ ...
        │
        ▼
┌─ Proxy API Key Middleware ────────────────────────┐
│  1. proxy_auth.enabled == false? → pass through   │
│  2. Extract Basic Auth credentials                │
│  3. Password == global_token? → allow + log       │
│  4. SHA-256(password) matches api_keys row?        │
│     → allow + set UserInfo in context + log       │
│  5. No match → 401 Unauthorized                   │
└───────────────────────────────────────────────────┘
        │
        ▼
   Adapter.ServeHTTP (PyPI, npm, ...)
```

**Transport: HTTP Basic Auth.** Password = API klíč (PAT nebo globální token). Username je libovolný — ignoruje se pro autentikaci, loguje se do auditu. Dokumentace doporučí: "use any value, e.g. your email for audit trail".

**Proč Basic Auth:**
- Funguje nativně ve všech package managerech bez workaroundu
- `pip install --index-url https://user:TOKEN@host/simple/`
- `npm config set //host:4873/:_authToken TOKEN`
- `docker login host -u user -p TOKEN`
- `nuget sources add -Source URL -UserName user -Password TOKEN`
- `gem sources -a https://user:TOKEN@host:8086/`
- `GONOSUMCHECK=* GOPROXY=https://user:TOKEN@host:8087 go get ...`

### Změny v servisní vrstvě

#### Nový middleware: `internal/auth/apikey.go`

```go
type APIKeyMiddleware struct {
    db           *config.GateDB
    globalToken  string  // plaintext global token (from env), empty = disabled
}

func NewAPIKeyMiddleware(db *config.GateDB, globalToken string) *APIKeyMiddleware

// Authenticate returns http.Handler middleware.
// Extracts Basic Auth password, validates against global token or DB lookup.
// On success, stores identity in request context via ContextWithUser().
func (m *APIKeyMiddleware) Authenticate(next http.Handler) http.Handler
```

Middleware flow:
1. `proxy_auth.enabled == false` → middleware se nevkládá vůbec (nulový overhead)
2. Extrakce Basic Auth credentials z `Authorization` headeru (pouze `Basic` schéma; `Bearer` a jiné se ignorují → 401)
3. Pokud `password == globalToken` → allow, `UserInfo{Email: username, Name: "global-token"}`
4. Pokud `SHA-256(password)` matchuje enabled řádek v `api_keys` (`WHERE key_hash = ? AND enabled = 1`) → allow, `UserInfo{Email: owner_email, Name: name}`
5. Jinak → `401 Unauthorized` s `WWW-Authenticate: Basic realm="shieldoo-gate"` headerem

**Constant-time comparison** pro globální token (`crypto/subtle.ConstantTimeCompare`).

**Secret logging ochrana:** Middleware NESMÍ logovat hodnotu `Authorization` headeru ani extrahovaný password. Při auth failure se loguje pouze: IP, username (z Basic Auth), a výsledek (denied). Přidáme unit test `TestAPIKeyMiddleware_DoesNotLogSecret`.

### Konfigurace

Nová sekce `proxy_auth` (oddělená od stávající `auth` sekce pro OIDC):

```yaml
# API key authentication for proxy endpoints (pip, npm, docker, etc.)
# When disabled (default), proxy endpoints are open — protected by network segmentation.
# When enabled, all proxy requests must include a valid API key via HTTP Basic Auth.
proxy_auth:
  enabled: false
  global_token_env: "SGW_PROXY_TOKEN"    # env var holding the global shared token
```

Config struct:

```go
type ProxyAuthConfig struct {
    Enabled        bool   `mapstructure:"enabled"`
    GlobalTokenEnv string `mapstructure:"global_token_env"`
}
```

**Tři scénáře:**

| `proxy_auth.enabled` | `global_token_env` set? | OIDC `auth.enabled`? | Chování |
|----------------------|------------------------|---------------------|---------|
| `false` | — | — | Proxy otevřené (default) |
| `true` | ano | ne | Pouze globální token funguje (žádné PAT — admin API je otevřené, nemá smysl) |
| `true` | ano | ano | Globální token + per-user PAT (plný režim) |

### TLS / Reverse proxy

**Basic Auth posílá credentials v base64 (ne šifrovaně) — TLS je povinný pokud je proxy_auth enabled.**

Shieldoo Gate sám TLS neterminuje — spoléhá na reverse proxy nebo ingress controller. Toto je záměr (separation of concerns, certificate management patří do infrastruktury).

#### Kubernetes (Ingress)

Všechny protokoly (PyPI, npm, NuGet, Maven, RubyGems, Go Modules) jsou standardní HTTP — fungují za jakýmkoliv Ingress controllerem (nginx, Traefik, Envoy) bez problémů.

**Docker je speciální případ:** Docker registry protocol je HTTP, ale Docker klient vyžaduje TLS pro jakýkoliv registry kromě `localhost`. Ingress musí terminovat TLS a forwardovat na Docker port.

Příklad Ingress (nginx):

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: shieldoo-gate
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: "0"       # unlimited upload for docker push
    nginx.ingress.kubernetes.io/proxy-read-timeout: "600"   # long timeout for large images
spec:
  tls:
    - hosts: ["gate.example.com"]
      secretName: gate-tls
  rules:
    - host: gate.example.com
      http:
        paths:
          - path: /simple
            pathType: Prefix
            backend:
              service:
                name: shieldoo-gate-pypi
                port: { number: 5000 }
          - path: /v2
            pathType: Prefix
            backend:
              service:
                name: shieldoo-gate-docker
                port: { number: 5002 }
          # ... analogicky pro npm (/), nuget (/v3), maven (/), rubygems (/), gomod (/)
```

**Alternativa:** Jeden port per protokol → jeden Ingress per port (jednodušší, bez path-based routing). Každý Service dostane vlastní hostname (pypi.gate.example.com, npm.gate.example.com, ...).

#### Docker Compose (lokální vývoj / malé deploymenty)

Pro docker-compose se přidá Caddy (nebo nginx) jako reverse proxy s automatickým TLS (Let's Encrypt nebo self-signed):

```yaml
services:
  caddy:
    image: caddy:2.9.1-alpine
    ports:
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile:ro
      - caddy-data:/data
    depends_on:
      - shieldoo-gate

  shieldoo-gate:
    # ... existující konfigurace, porty NEEXPONOVANÉ ven
    expose:
      - "5000"   # PyPI
      - "4873"   # npm
      - "5001"   # NuGet
      - "5002"   # Docker
      - "8080"   # Admin API
      - "8085"   # Maven
      - "8086"   # RubyGems
      - "8087"   # Go Modules
```

Caddyfile:

```
pypi.gate.localhost {
    reverse_proxy shieldoo-gate:5000
}
npm.gate.localhost {
    reverse_proxy shieldoo-gate:4873
}
docker.gate.localhost {
    reverse_proxy shieldoo-gate:5002
}
nuget.gate.localhost {
    reverse_proxy shieldoo-gate:5001
}
admin.gate.localhost {
    reverse_proxy shieldoo-gate:8080
}
maven.gate.localhost {
    reverse_proxy shieldoo-gate:8085
}
rubygems.gate.localhost {
    reverse_proxy shieldoo-gate:8086
}
gomod.gate.localhost {
    reverse_proxy shieldoo-gate:8087
}
```

**Pro lokální dev bez TLS:** Pokud `proxy_auth.enabled: false`, TLS není nutný. Pokud je zapnutý a běží to bez TLS, startup warning v logu: "proxy_auth is enabled but no TLS detected — credentials will be sent in plaintext".

#### Kompatibilita protokolů s reverse proxy

| Protokol | HTTP metody | Streaming? | Velké body? | Speciální požadavky |
|----------|------------|------------|-------------|---------------------|
| PyPI (PEP 503) | GET | ne | ne | žádné |
| npm | GET, PUT | ne | balíčky (~MB) | žádné |
| NuGet | GET | ne | ne | žádné |
| Docker v2 | GET, HEAD, PUT, PATCH, DELETE | ano (blob upload) | velké image layers (~GB) | `proxy-body-size: 0`, chunked transfer |
| Maven | GET | ne | ne | žádné |
| RubyGems | GET | ne | ne | žádné |
| Go Modules | GET | ne | ne | žádné |

Všechny protokoly jsou standardní HTTP/1.1 — žádný nevyžaduje WebSocket, gRPC, ani HTTP/2. Reverse proxy kompatibilita je 100%.

### Docker specifika

Docker client používá `/v2/` token auth flow. Middleware musí fungovat i pro Docker adapter:
- Docker klient posílá Basic Auth při `docker login`
- Pokud middleware vrátí 401, měl by vracet `WWW-Authenticate: Basic realm="shieldoo-gate"` header, který Docker klient rozumí

## Rizika a mitigace

| Riziko | Dopad | Pravděpodobnost | Mitigace |
|--------|-------|-----------------|----------|
| Timing side-channel na hash lookup | Nízký — útočník potřebuje hodně pokusů | Nízká | `subtle.ConstantTimeCompare` pro globální token; DB lookup je inherentně constant-time (index scan) |
| Uniknutý klíč v .pypirc / .npmrc | Střední | Střední | `sgw_` prefix → GitHub secret scanning; revoke endpoint; audit log |
| DB zátěž na každý request (hash lookup) | Nízký | Nízká | Jeden SELECT s indexed UNIQUE sloupcem; optional in-memory cache v budoucnu |
| Docker /v2/ token flow | Střední — Docker má specifický auth protocol | Střední | Basic Auth funguje přímo s `docker login`; middleware vrací správný `WWW-Authenticate` header |
| Breaking change pro existující uživatele | Vysoký | Žádná | `proxy_auth.enabled` default `false` — zpětná kompatibilita |
| Credentials v plaintext bez TLS | Vysoký — token sniffing | Střední | Startup warning pokud `proxy_auth.enabled` bez TLS; dokumentace vyžaduje reverse proxy s TLS |

## Poznámky

- **Zpětná kompatibilita:** `proxy_auth.enabled: false` je default — žádná změna chování pro existující instalace
- **Oddělení od OIDC:** `proxy_auth` je nezávislá config sekce. OIDC chrání admin API, proxy_auth chrání proxy endpointy. Jsou ortogonální.
- **Username v Basic Auth:** Ignorován pro autentikaci. Logován do auditu. Dokumentace doporučí používat email nebo popis účelu (např. "ci-bot").
- **`last_used_at` update:** Debounce přes in-memory `sync.Map[keyID → timestamp]` s periodic flush každých 30 sekund. Kolaps N requestů na 1 UPDATE per key per interval. Flush i při graceful shutdown. Žádné unbounded goroutines.
- **TLS je povinný při zapnutém proxy_auth.** Gate sám TLS neterminuje — vyžaduje reverse proxy (Caddy/nginx/Ingress). Startup warning při absenci TLS.
- **Revokace je permanentní:** `DELETE` nastaví `enabled=false`, klíč nelze obnovit. Chcete nový klíč → vytvořte nový. Žádný re-enable endpoint.
- **Config validace `global_token_env`:** Pokud env var není nastavená, loguje se warning (analogicky k `validateAlerts`).
- **Audit trail u globálního tokenu:** Username z Basic Auth je kontrolovaný klientem — audit trail je nespolehlivý (klient může poslat cokoliv). Toto je známé omezení globálního tokenu; pro spolehlivý audit použijte per-user PAT.
- **Known gap — rate limiting:** Brute-force ochrana na 401 responses není v MVP. Riziko je nízké díky vysoké entropii klíčů (256 bit). Per-IP rate limiting bude follow-up task.
- **Budoucí rozšíření:** Expiry enforcement na klíčích (sloupec již v DB), per-key scope (omezení na ecosystem), rate limiting per key. Záměrně vynecháno — YAGNI.
