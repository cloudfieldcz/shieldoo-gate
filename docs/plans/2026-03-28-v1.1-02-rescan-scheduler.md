# v1.1 Rescan Scheduler — Technická analýza

## Popis

Implementace generického rescan scheduleru, který periodicky prochází cachované artefakty a znovu je skenuje. Cílem je detekovat nově objevené hrozby v dříve považovaných za čistých artefaktech (zero-day window closure).

### Proč

- Zero-day window: mezi prvním stažením artefaktu a aktualizací threat feedu/scanner DB může uplynout hodiny až dny
- Současný stav: pouze Docker sync (`sync.go`) re-skenuje — ale jen Docker image manifesty, ne PyPI/npm/NuGet artefakty
- Manual rescan existuje (API endpoint), ale žádný automatický scheduler
- `rescan_due_at` sloupec v `artifact_status` tabulce existuje, ale není využíván (kromě Docker sync)
- `initial-analyse.md:698-731` definuje scheduler s priority queue a batch processing

### Out of Scope

- Docker tag sync (již implementováno v `sync.go`, tento scheduler je pro non-Docker artefakty a pro generický rescan)
- Rescan triggery z threat feed updatu (`on_feed_update`) — explicitně odloženo do Fáze 2. Původní spec (`initial-analyse.md:513`) definuje `on_feed_update: true`, ale implementace vyžaduje callback z `threatfeed.Client.Refresh()` → trigger rescan cycle.
- UI pro scheduler konfiguraci

### Prerekvizity identifikované při review

- **`last_accessed_at` se nikdy neaktualizuje** — `artifacts.last_accessed_at` je nastaveno pouze při INSERT (`base.go:162`). Žádný `UPDATE last_accessed_at` neexistuje v cache hit path adaptérů. Priority ordering `ORDER BY last_accessed_at DESC` tak vrací artefakty seřazené podle času prvního stažení, ne podle skutečné frekvence použití. **Nutná prerekvizita:** přidat `UPDATE artifacts SET last_accessed_at = ? WHERE id = ?` do cache-hit path každého adaptéru.
- **`InsertScanResults` je N+1 write pattern** — `adapter/base.go:116-140` vkládá scan výsledky jeden po jednom v loopu bez transakce. Při batch_size=100 a 5 scanerech = 500 individuálních INSERT per cyklus. **Doporučení:** zabalit do jedné transakce.
- **Chybí index na `artifacts.last_accessed_at`** — žádný index v `001_init.sql:57-60`. Nutný pro efektivní ORDER BY.

## Návrh řešení

### Architektura

```
                    ┌─────────────────────────────┐
                    │      RescanScheduler         │
                    │                              │
ticker (interval)──▶│  1. Query: artifact_status   │
                    │     WHERE rescan_due_at < NOW │
                    │     OR status = PENDING_SCAN  │
                    │     ORDER BY last_accessed_at  │
                    │     LIMIT batch_size          │
                    │                              │
                    │  2. Per artifact:             │
                    │     ├── Load from cache       │
                    │     ├── ScanAll()            │
                    │     ├── Evaluate()           │
                    │     ├── Update status        │
                    │     ├── WriteAuditLog        │
                    │     └── Set next rescan_due  │
                    └─────────────────────────────┘
```

**Klíčová rozhodnutí:**

1. **`time.NewTicker`, ne cron library** — stejný pattern jako Docker sync a threat feed. Jednodušší, žádná nová závislost. Scheduler běží v intervalu (default `6h`), zpracuje batch artefaktů.
2. **Priority queue přes SQL** — `ORDER BY last_accessed_at DESC` (nejpoužívanější první). **Prerekvizita:** adaptéry musí aktualizovat `last_accessed_at` na cache-hit (dosud se neděje). Bez toho je řazení dle cached_at (první stažení).
3. **Batch processing** — zpracuje max `batch_size` (default 100) artefaktů per cyklus. Prevence přetížení scannerů.
4. **Semaphore concurrency** — max `max_concurrent` (default 5) paralelních skenů. Stejný pattern jako Docker sync.
5. **Fail-open** — scan error → status zůstává CLEAN, loguje se warning. Konzistentní s celkovým design principem.
6. **Rescan interval per-artifact** — po dokončení rescanu se `rescan_due_at` nastaví na `now + rescan_interval`. Artefakty s vyšší frekvencí přístupu se rescanují častěji (implicitně přes priority).
7. **Změna config formátu** — původní spec (`initial-analyse.md:512`) používá cron syntax (`"0 2 * * *"`), tato analýza přechází na Go duration (`"6h"`). Důvod: žádná nová závislost (robfig/cron). Kandidát na ADR.

### Query pro výběr artefaktů k rescanu

```sql
SELECT a.id, a.ecosystem, a.name, a.version, a.storage_path, a.sha256, a.size_bytes, a.upstream_url
FROM artifacts a
JOIN artifact_status s ON a.id = s.artifact_id
WHERE s.status = 'PENDING_SCAN'
   OR (s.status IN ('CLEAN', 'SUSPICIOUS') AND s.rescan_due_at IS NOT NULL AND s.rescan_due_at < ?)
ORDER BY
    CASE WHEN s.status = 'PENDING_SCAN' THEN 0 ELSE 1 END,  -- PENDING_SCAN first
    a.last_accessed_at DESC                                    -- most used next
LIMIT ?
```

Priority:
1. `PENDING_SCAN` (manuální rescan z API) — okamžitě
2. `CLEAN` / `SUSPICIOUS` s `rescan_due_at < now` — seřazené od nejpoužívanějších

`QUARANTINED` artefakty se NE-rescanují automaticky (require manual release).

### Rescan flow per artifact

```go
func (s *RescanScheduler) rescanArtifact(ctx context.Context, art artifactRow) error {
    // 1. Verify artifact still exists in cache
    localPath, err := s.cache.Get(ctx, art.ID)
    if err == cache.ErrNotFound {
        // Artifact evicted from cache — clear rescan_due_at, skip
        return s.clearRescanDue(art.ID)
    }

    // 2. Build scanner.Artifact
    scanArtifact := scanner.Artifact{
        ID: art.ID, Ecosystem: ..., Name: ..., Version: ...,
        LocalPath: localPath, SHA256: art.SHA256, ...
    }

    // 3. Scan (fail-open: preserve current status on error)
    results, scanErr := s.scanEngine.ScanAll(ctx, scanArtifact)
    if scanErr != nil {
        log.Error().Err(scanErr).Str("artifact", art.ID).Msg("rescan: scan error, preserving current status")
        // DŮLEŽITÉ: NEpředávat nil results do policy engine — mohlo by to
        // revertovat SUSPICIOUS artefakt zpět na CLEAN. Při scan error
        // pouze aktualizovat rescan_due_at a přeskočit status update.
        s.updateRescanDueAt(art.ID, time.Now().Add(s.rescanInterval))
        return nil
    }

    // 4. Policy evaluation (pouze pokud scan proběhl úspěšně)
    decision := s.policyEng.Evaluate(ctx, scanArtifact, results)

    // 5. Update status based on decision
    switch decision.Action {
    case policy.ActionQuarantine:
        // Set QUARANTINED, write audit log, trigger alert
    case policy.ActionBlock:
        // Set QUARANTINED (block from rescan = quarantine), write audit log
    default:
        // Set CLEAN, update rescan_due_at = now + rescanInterval
    }

    // DŮLEŽITÉ: Kroky 5-7 (status update + scan results + audit log)
    // by měly být v jedné transakci, aby nedošlo k partial update
    // při selhání uprostřed flow.

    // 6. Persist scan results
    adapter.InsertScanResults(s.db, art.ID, results)

    // 7. Write audit log
    adapter.WriteAuditLog(s.db, model.AuditEntry{
        EventType:  model.EventScanned,
        ArtifactID: art.ID,
        Reason:     fmt.Sprintf("scheduled rescan: %s", decision.Reason),
    })

    return nil
}
```

### Inicializace rescan_due_at pro existující artefakty

Při prvním spuštění scheduleru budou existující artefakty mít `rescan_due_at = NULL`. Scheduler je ignoruje (WHERE clause filtruje `rescan_due_at IS NOT NULL AND rescan_due_at < ?`). Pro existující artefakty se `rescan_due_at` nastaví při:
- Prvním úspěšném rescanu (scheduler je vybere přes `PENDING_SCAN` status)
- Nebo přidáme jednorázovou migraci/bootstrap, která nastaví `rescan_due_at = cached_at + rescan_interval` pro všechny CLEAN artefakty

**Doporučení:** Bootstrap v migraci — jednodušší, deterministic:

```sql
-- Migration: initialize rescan_due_at for existing CLEAN artifacts
UPDATE artifact_status
SET rescan_due_at = datetime(
    (SELECT cached_at FROM artifacts WHERE artifacts.id = artifact_status.artifact_id),
    '+24 hours'
)
WHERE status = 'CLEAN' AND rescan_due_at IS NULL;
```

**Pozor:** Tato migrace používá SQLite-specifickou funkci `datetime()`. Pokud je implementován v1.1-04 (PostgreSQL), potřebuje dual-dialect verzi:
- SQLite: `datetime(cached_at, '+24 hours')`
- PostgreSQL: `cached_at + INTERVAL '24 hours'`

### Integrace s alerting systémem

Rescan scheduler volá `adapter.WriteAuditLog()` → automaticky triggeruje alert přes hook (z v1.1 alerts analýzy). Při quarantine ze scheduled rescanu:

## Akceptační kritéria

1. Scheduler periodicky (dle `schedule` config) prochází artefakty s `rescan_due_at < now` nebo `status = PENDING_SCAN`.
2. `PENDING_SCAN` artefakty (z manual rescan API) mají prioritu před scheduled rescany.
3. Po rescanu se `rescan_due_at` aktualizuje na `now + rescan_interval`.
4. Artefakt reklasifikovaný jako MALICIOUS/SUSPICIOUS se karanténuje a audit log + alert se zapíší.
5. Scan error NEZMĚNÍ status artefaktu — zůstává v předchozím stavu. Nil results se NEpředávají do policy engine.
6. Max `batch_size` artefaktů se zpracuje za cyklus.
7. Max `max_concurrent` paralelních skenů běží současně.
8. Artefakt eviktovaný z cache se přeskočí a `rescan_due_at` se vynuluje.
9. `QUARANTINED` artefakty se NE-rescanují automaticky.
10. Po `POST /api/v1/artifacts/{id}/rescan` je artefakt rescanován nejpozději v příštím scheduler cyklu.
11. Docker artefakty mohou být volitelně vyloučeny ze scheduleru pokud Docker sync je enabled (prevence duplicitního scanu).

## Rizika a mitigace

| Riziko | Dopad | Pravděpodobnost | Mitigace |
|--------|-------|-----------------|----------|
| Scanner overload při velkém batch | Střední — degradace scan performance pro nové requesty | Střední | Semaphore (max_concurrent), batch_size limit, scanner timeout |
| SQLite lock contention při rescanu | Střední — latence pro live requesty | Střední | Batch processing (ne všechno najednou), WAL mode již zapnut |
| Cache eviction race | Nízký — artefakt smazán mezi query a Get() | Nízká | Graceful handling: cache.ErrNotFound → skip + clear rescan_due_at |
| Rescan nedetekuje novou hrozbu | Nízký — scanner DB neaktualizovaná | Střední | on_feed_update flag (budoucí), threat feed checker vždy aktuální |
| Fail-open revertuje SUSPICIOUS na CLEAN | Vysoký — silent security demotion | Střední | Při scan error zachovat current status, nepředávat nil do policy engine |
| Multi-instance race (budoucí PostgreSQL HA) | Střední — duplicitní rescan | Nízká | `SELECT ... FOR UPDATE SKIP LOCKED` (PostgreSQL); SQLite = single instance |
| `last_accessed_at` není aktualizován | Vysoký — priority ordering nefunkční | Jistý (pre-existing) | Prerekvizita: přidat UPDATE do cache-hit path adaptérů |

## Poznámky

- **Vztah k Docker sync:** Docker sync (`sync.go`) a RescanScheduler jsou nezávislé. Docker sync kontroluje upstream digest changes + rescanuje Docker manifesty. RescanScheduler rescanuje všechny cachované artefakty (včetně Docker, pokud mají `rescan_due_at`). Overlap je možný ale neškodný — duplicitní scan je idempotentní.
- **on_feed_update:** Config field existuje ale zatím nepodporován. Implementace by vyžadovala callback z `threatfeed.Client.Refresh()` → trigger full rescan cycle. Explicitně odloženo do Fáze 2. Viz Out of Scope.
- **Zpětná kompatibilita:** 100% — scheduler je opt-in (`enabled: true`). Existující artefakty dostanou `rescan_due_at` přes migraci.
- **Docker sync overlap:** Docker sync a RescanScheduler mohou rescanovat stejný artefakt. Toto je bezpečné (idempotentní) ale neefektivní. Zvážit vyloučení `ecosystem='docker'` z scheduler query pokud Docker sync je enabled.
- **Dokumentační deliverables:** Aktualizovat `docs/index.md` (link na scheduler docs), `docs/configuration.md`. ADR kandidát: rozhodnutí použít `time.NewTicker` místo cron library.
